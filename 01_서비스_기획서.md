# 영업 관리 프로그램 - 서비스 기획서

> 영업직 개인을 위한 영업 데이터 관리 SaaS

---

## 1. 서비스 정체성

### 1.1 한 줄 정의
**영업사원이 노트북과 핸드폰을 오가며 영업 데이터를 동기화하고 관리할 수 있는 개인용 영업 관리 도구**

### 1.2 타겟
- **사용자**: 영업직 개인 (B2C)
- **외관**: B2B처럼 보이지만, 회사가 도입하는 게 아니라 영업사원 개인 대상
- **시장**: 한국 영업사원

### 1.3 풀고자 하는 문제 (실제 페인포인트)
1. 노트북에서 기록한 영업 데이터를 운전 중/외근 중 핸드폰에서 바로 못 봄
2. 거래처 데이터가 엑셀, 명함, 노트, 메신저에 흩어져 있음
3. 받은 명함이 책상 서랍에서 잠자고 있음
4. 영업 관리를 위한 CRM은 대부분 회사용이라 개인이 쓰기 무거움
5. 영업 기록할 때 상품 가격을 다른 파일에서 찾아 옮겨 적어야 함

### 1.4 정체성 원칙 (핵심 가치관)
- **메인은 "영업 관리"** — 캘린더, 회의 녹음, 견적서 같은 부가 기능은 빼거나 후순위
- **완성도가 낮은 기능은 만들지 않는다** — "고깃집의 김치찌개" 원칙: 60%만 흉내내는 기능은 서비스 신뢰도를 떨어뜨림
- **선택지를 많이 준다고 좋은 게 아니다** — 핵심에 집중
- **정체성 흔들리는 부가 기능은 거절** — "해보자"가 아니라 "이게 핵심을 해결하는가"

### 1.5 차별점
- 회사 양식에 종속되지 않는 개인용
- 노트북-모바일 진짜 동기화 (운전 중에도 바로 보임)
- 명함 스캔으로 거래처 자동 등록
- Off-the-record 메모 (예: "삼성 반도체 라인 DX사업부 OO부서에 OO 있음" 같은 비공식 인사이트)

## 2. 핵심 기능 (Must-Have)

### 2.1 영업 데이터 관리 (CRUD)
- 영업에 필요한 모든 데이터 저장
- **멀티 디바이스 동기화**: 온라인/오프라인/PC/모바일 모두 동일하게 보여야 함
  - 노트북에서 기록 → 운전 중 핸드폰에서 바로 확인
- 수동 입력 지원

### 2.2 데이터 입출력
| 기능 | 설명 |
|------|------|
| **엑셀 Import** | 다른 곳에서 작성한 엑셀을 가져와서 등록 |
| **엑셀 Export** | 데이터를 엑셀로 추출 |
| **PDF Export** | 데이터를 PDF로 추출 |

### 2.3 필터링 시스템
- **컬럼 기반 필터링**: 예) 지역 컬럼에서 "서울"인 사람만 보기
- **탭 기반 주요 필터**: 자주 쓰는 필터는 아예 탭으로 분리
  - 예: "이번 달 구매 가능성 높은 사람" 탭
  - 예: "지역별" 탭

### 2.4 거래처 정보 관리 (디지털 명함첩)
- **명함 스캔 → 자동 등록 (OCR)**
  - 사진 찍으면 DB에 자동 저장
  - 이름 검색하면 "어디 회사 누구" 바로 조회
- **수동 입력**도 가능
- **회사 정보 추가 입력 가능** (선택)
- **Off-the-record 메모**:
  - 거래처/회사별 자유 메모 탭
  - 비공식 인사이트, 인맥 정보 등

### 2.5 인증
- 회원가입 / 로그인
- 사용자별 데이터 분리

## 3. 부가 기능 (후순위 또는 보류)

### 3.1 후순위 (Phase 4 이후)
| 기능 | 우선순위 |
|------|----------|
| 상품 카탈로그 + 자동 입력 (예: "바나나" 치면 가격 자동) | 4 |
| 알림 | 4 |
| 데이터 백업/복구 | 4 |
| 명함 OCR (Phase 3에 포함 가능) | 3~4 |

> 수익화/구독/결제는 위 우선순위에서 제외. 앱 출시 후 별도 단계로 진행.

### 3.2 보류/제외 결정
| 기능 | 사유 |
|------|------|
| ❌ 견적서 생성 | 회사 양식과 맞추기 어려움 |
| ❌ 회의 녹음 | 전화 끊김 등 구조적 완성도 낮음 |
| ❌ 결제 서류 자동 생성 | 회사 양식 매칭 불가 |
| ⚠️ 캘린더/일정/메모 | 정체성 흐림. 캘린더는 본인이 쓰던 앱 사용. 사용자 요청 누적되면 재검토 |

### 3.3 보류 결정의 근거
- 핵심 기능이 아니면 신뢰도만 떨어뜨림
- 사용자가 원하는 결과물의 60%만 흉내내는 기능은 안 만드는 게 나음
- 정체성을 흔드는 기능은 우리도 사용자도 헷갈리게 만듦

## 4. 단계별 로드맵

### Phase 0: 제품 뼈대 (Day 1~2)
- 로그인 / 회원가입
- 기본 레이아웃
- 모바일 대응 (PWA)
- 사용자별 데이터 분리 (RLS 또는 user_id 기반)
- /health API

### Phase 1: MVP 핵심 (Week 1~2)
- 거래처(Customer) 등록/수정/삭제
- 담당자/영업건(Deal) 등록/수정/삭제
- 영업 기록
- 지역/담당자/상태/가능성/예상 월 기준 필터
- 검색
- 엑셀 Export
- **PWA + IndexedDB 오프라인 조회 (Week 2 후반)**

### Phase 2: 입출력 + 사용자 피드백 반영 (Week 3)
- 엑셀 Import
- 자주 쓰는 필터 탭 저장
- "이번 달 계약 가능성 높은 건" 탭
- "지역별" 탭
- 모바일 입력 UX 개선
- 데이터 일괄 수정
- PDF 요약 출력

### Phase 3: 거래처 + 명함 (Week 4)
- 거래처 회사 정보 (Company)
- 명함 스캔 OCR (Naver CLOVA OCR)
- Off-the-record 메모 탭
- 베타 사용자 5~10명 모집
- 피드백 수집 → 다음 우선순위 결정

### Phase 4: 확장 기능 (필요 시, 앱 출시 후 베타 피드백 보고 결정)
- 상품/단가 DB
- 상품 검색 후 영업 기록 자동 입력
- 알림 시스템
- 데이터 백업/복구
- 고급 인사이트/분석

> **수익화/구독/결제 시스템은 별도 단계로 분리. 앱이 충분히 안정화되고 사용자 검증이 완료된 후 별도 기획.**

## 5. 데이터 모델 (초안)

### 5.1 주요 엔티티

```
users (Supabase Auth)
  └─ id, email, ...

companies (회사 정보)
  ├─ id, user_id (FK)
  ├─ name, industry, address
  └─ created_at

customers (거래처 = 사람)
  ├─ id, user_id (FK)
  ├─ company_id (FK, nullable)
  ├─ name, phone, email
  ├─ position, region        ← 필터 핵심 컬럼
  ├─ business_card_url       ← Supabase Storage URL
  ├─ ocr_raw_text            ← OCR 원본 (재파싱용)
  └─ created_at, updated_at

deals (영업 건)
  ├─ id, user_id (FK)
  ├─ customer_id (FK)
  ├─ title, status (lead/in_progress/won/lost)
  ├─ amount, probability (0~100)
  ├─ expected_close_date
  ├─ notes
  └─ created_at, updated_at

notes (off-the-record 메모)
  ├─ id, user_id (FK)
  ├─ customer_id or company_id (FK)
  ├─ content, tags[]
  └─ created_at

products (상품 카탈로그, Phase 5)
  ├─ id, user_id (FK)
  ├─ name, price, unit
  └─ created_at
```

### 5.2 데이터 격리
- 모든 테이블에 `user_id` 컬럼
- Supabase RLS 정책: `user_id = auth.uid()` 강제
- 백엔드에서도 모든 쿼리에 `user_id` 필터 강제

## 6. 핵심 사용자 시나리오

### 시나리오 1: 운전 중 핸드폰 조회
1. 사용자가 어제 노트북에서 거래처 "송재근" 정보 업데이트
2. 다음 날 운전 중 핸드폰에서 앱 열기
3. 오프라인 상태에서도 "송재근" 검색 → 정보 표시 (IndexedDB 캐시)
4. 통화 후 새 메모 추가 → 온라인 복귀 시 자동 동기화

### 시나리오 2: 명함 스캔
1. 거래처 미팅 후 명함 받음
2. 앱에서 카메라로 명함 촬영
3. CLOVA OCR이 자동 파싱 → 이름/전화/회사/직급 추출
4. 확인 화면에서 수정 가능
5. 저장 → 거래처 리스트에 자동 추가

### 시나리오 3: 엑셀로 일괄 등록
1. 기존에 엑셀로 관리하던 거래처 100명
2. 앱에서 엑셀 업로드
3. 컬럼 매핑 (엑셀 컬럼 → 앱 필드)
4. 검증 후 일괄 등록

### 시나리오 4: 월말 보고용 PDF
1. 이번 달 영업 건 필터링
2. PDF Export 버튼
3. 영업 리스트가 정리된 PDF 다운로드
4. 회사에 제출 (회사 양식이 아닌, 본인 정리용)

## 7. 의사결정 기록

### 7.1 SEO 불필요
- 로그인 후 사용하는 서비스
- → SSR 필요 없음
- → React + Vite로 충분

### 7.2 영업사원 개인 대상 B2C
- 회사 도입이 아니므로 회사 양식 강제 불가
- → 견적서/결제 서류 자동 생성 제외
- → 개인의 자유로운 메모 중심

### 7.3 완성도 우선 원칙
- 60% 완성도 기능은 만들지 않음
- 핵심에 집중

### 7.4 사용자 피드백 기반 고도화
- 최대한 빠르게 배포해서 실제 사용자 반응 받기
- 4주 안에 Phase 3까지 배포
- Phase 4(확장 기능) 이후는 베타 피드백 보고 결정
- 수익화/구독/결제는 별도 단계로 분리 (현재 로드맵에서 제외)
