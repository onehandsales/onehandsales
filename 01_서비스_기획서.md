# 서비스 기획서

> 영업사원의 머릿속을 정리해주는 도구
>
> 작성일: 2026-05-27
> 서비스명: **(미정)** — 임시 코드네임 사용

---

## 1. 한 줄 정의

영업직 개인이 고객·딜·거래처·제품 데이터를 한 곳에 쌓고, PC와 모바일에서 동일하게 조회·관리할 수 있는 **개인용 영업 관리 도구**.

---

## 2. 왜 만드는가

영업사원은 하루에도 수십 명을 상대한다. 전화하고, 미팅하고, 견적 보내고, 다시 전화한다.
그 과정에서 쌓이는 정보들 — 누가 뭘 원하는지, 언제 결정이 나는지, 이 사람한테 어떻게 접근해야 하는지 — 이게 **영업사원의 진짜 자산**이다.

그런데 그 자산이 지금 어디 있는지 물으면 대부분 이렇게 답한다.

> "노트북 엑셀에 있는데요."
> "명함은 서랍에 있고요."
> "이 사람 메모는 카카오톡 나에게 보내기에 있어요."

**흩어져 있다. 그리고 현장에서 꺼내 볼 수가 없다.**

고객사 주차장에서 미팅 직전, "이 사람이 지난번에 뭐라고 했더라"를 확인하려는데 그 메모는 집 책상 노트북 안에 있다. 회사 CRM은 팀장 보고용이고, 내가 발로 뛰며 쌓은 감과 정보는 시스템 어디에도 내 것으로 남지 않는다.

이 서비스는 이 문제를 푼다.

---

## 3. 누구를 위한 서비스인가

회사가 도입하는 툴이 아니다. **영업사원 개인이 자기 돈 내고 쓰는 개인 도구**다.

### 3.1 타겟
- **1차 타겟**: **계측기 기술영업사원**
- **확장 타겟**: B2B 기술영업 전반 (산업재, 의료기기, 화학약품 등)
- 거래처 수십 곳을 동시에 관리하면서 각각의 상황과 맥락을 기억해야 하는 사람

### 3.2 왜 기존 CRM은 안 되는가
- **너무 무겁다**: 팀 관리, 파이프라인 리포트, 권한 설정 — 혼자 쓰는 사람에게 불필요
- **회사 종속적**: 내가 쌓은 데이터를 회사를 떠나면 가져갈 수 없음

### 3.3 핵심 약속
**어떤 회사를 다니든, 회사를 옮기든, 내가 쌓은 거래처 정보와 영업 이력은 내 계정에 남는다.**

---

## 4. 핵심 페인포인트 (직접 풀 문제)

1. 노트북에 기록한 데이터를 이동 중 모바일에서 못 봄
2. 거래처 정보가 명함·엑셀·메신저에 흩어져 있음
3. 영업 기록할 때 제품 단가를 다른 파일에서 찾아야 함
4. 기존 CRM은 회사용이라 개인이 쓰기 무거움
5. 회사 시스템에 못 쓰는 **비공식 인사이트**를 둘 곳이 없음

---

## 5. 어떤 가치를 주는가

### 5.1 현장에서 바로 꺼내볼 수 있다
노트북에서 입력한 내용이 5초 후 핸드폰에서 보인다. 운전 중에, 고객사 로비에서, 엘리베이터 안에서 — 인터넷이 없어도 마지막으로 본 데이터는 그대로 남아 있다.

### 5.2 공식 기록에 남길 수 없는 정보를 담는다
"이 라인 실제 결정권은 옆 부서 팀장이 한다", "이 사람은 직접 연락하면 싫어한다", "이 회사 예산은 매년 10월에 풀린다" — 회사 시스템에는 절대 못 쓰지만 **영업에서 가장 중요한 정보**다. 이것을 안전하게 보관할 공간을 제공한다.

### 5.3 지금 뭐가 급한지 한눈에 보인다
이번 달 안에 결론 나는 건이 몇 개인지, 가능성이 높은 거래가 어디까지 왔는지 — 흩어진 데이터를 모아두면 자연스럽게 우선순위가 보인다. 엑셀에서는 볼 수 없던 그림이다.

### 5.4 명함이 데이터가 된다
받은 명함을 카메라로 찍으면 거래처 정보로 자동 저장된다. 서랍에서 잠자던 명함 뭉치가 검색 가능한 인맥 데이터베이스가 된다.

## 6. 도메인 구조

### 6.1 영업 관리 핵심 도메인 (8개)

```
User (사용자)
 ├── Company (회사)
 │    └── Contact (담당자)
 │         └── Deal (딜)
 │              ├── DealActivity (딜 활동 기록)
 │              └── Meeting (AI 회의록, nullable 연결)
 ├── Product (제품)
 └── Note (off-the-record 메모)
```

| 엔티티 | 역할 |
|--------|------|
| **User** | 서비스 계정. 모든 데이터는 User 단위로 완전 격리 |
| **Company** | 거래처 회사 (삼성전자, LG화학 등) |
| **Contact** | 거래처 담당자 개인. Company 소속 또는 독립 존재 가능 |
| **Deal** | 영업 건(기회). Contact 1명에 연결, Product 1개 참조(선택) |
| **DealActivity** | 딜에 쌓이는 활동 기록. 날짜 + 내용의 타임라인 |
| **Product** | 판매 제품 및 단가. 딜에 저장된 금액은 **입력 시점 고정** |
| **Note** | 비공식 메모. Company 또는 Contact에 연결 |
| **Meeting** | AI 회의록. 음성/텍스트 입력 → AI 요약. Deal에 nullable 연결 가능 |

### 6.2 결제 / 구독 도메인 (6개)

```
Plan ─ Subscription ─ Invoice ─ Payment ─ PaymentMethod
                                    │
                                    └── Refund
```

| 엔티티 | 역할 |
|--------|------|
| **Plan** | 요금제 (Free / Pro / Enterprise). 단가·기능 제한·트라이얼 기간 |
| **Subscription** | User별 구독 상태. 시작일/다음 결제일/상태 |
| **Invoice** | 청구서. 월별·연별 발행, 영수증용 |
| **Payment** | 결제 시도/완료 기록. 금액·상태·PG사 트랜잭션 ID |
| **PaymentMethod** | 결제 수단. PG사 토큰만 보관 (카드 정보 X) |
| **Refund** | 환불 기록. 부분/전액, 사유 |

**Subscription 상태 전이**:
```
trialing → active → (canceled OR past_due)
active → paused → active
active → canceled (즉시 또는 기간 종료 시)
past_due → active (결제 재시도 성공) OR canceled
```

**한국 법 준수**:
- 7일 이내 무조건 환불 (전자상거래법)
- 결제 수단은 PG사 토큰만 저장, 카드 정보 평문 저장 절대 금지

### 6.3 시스템 / 운영 도메인 (5개)

| 엔티티 | 역할 |
|--------|------|
| **AuditLog** | 관리자 변경 액션 추적 (법적/내부 분쟁 증거) |
| **Notification** | 발송된 알림 기록 (마감 3일 전/당일) |
| **NotificationPreference** | 사용자별 알림 on/off 설정 |
| **FeatureFlag** | 기능별 사용자 그룹 토글 (Pro 전용 기능 등) |
| **ImportJob** | 엑셀 Import 작업 기록 (성공/실패/에러 로그) |

### 6.4 파일 / 부가 도메인 (3개)

| 엔티티 | 역할 |
|--------|------|
| **BusinessCardImage** | 명함 원본 메타데이터 (Storage URL, OCR 상태, 신뢰도) |
| **FilterPreset** | 사용자 정의 탭/필터 저장 |
| **RefreshToken** | refresh 토큰 회전 추적 (Supabase가 일부 처리하므로 선택) |

### 6.5 도메인 총괄

| 영역 | 엔티티 수 |
|------|---------|
| 영업 관리 핵심 | 8개 (Meeting 포함) |
| 결제 / 구독 | 6개 |
| 시스템 / 운영 | 5개 |
| 파일 / 부가 | 3개 |
| **합계** | **22개** |

> 구체 컬럼/관계/인덱스는 DB 스키마 설계 단계에서 직접 진행. 이 문서는 **의미만 확정**.
> 결제 / 구독 도메인은 **2단계 내부 검증 완료 → 3단계 외부 공개** 전까지 구현 필수.

## 7. 정체성 원칙

### 7.1 의도적으로 좁다
견적서 자동 생성, 회의 녹음, 캘린더 관리, 팀 공유 — 이런 기능들을 **넣지 않는다**.

이미 잘 만들어진 다른 도구들이 있고, 어설프게 흉내 낸 기능이 오히려 신뢰를 깎는다. 이 서비스가 잘하는 것 하나에 집중한다.

> **"영업사원이 가진 정보를 한 곳에 모으고, 어디서든 꺼내볼 수 있게 하는 것."**

### 7.2 "고깃집의 김치찌개" 원칙
부가 기능의 완성도가 낮으면 서비스 전체 신뢰도가 떨어진다. **60% 완성도 기능은 만들지 않는다.**

### 7.3 안 만들 것 (명시적 제외)

| 기능 | 제외 이유 |
|------|----------|
| 견적서 자동 생성 | 회사마다 양식 달라서 완성도 보장 불가 |
| 회의 녹음 | 전화 끊김 등 구조적으로 완성도 낮음 |
| 캘린더 / 일정 관리 | 서비스 정체성 벗어남 |
| 팀 공유 / 협업 | 개인용 서비스. 팀 기능은 정체성 훼손 |
| 수익화 / 결제 | 서비스 검증 후 별도 기획 |

## 8. 시장 관점

- 기업용 CRM(Salesforce, HubSpot)은 팀·기업용 — 개인 영업사원을 위한 도구는 없음
- 한국 영업직 종사자 수백만 명 중 회사 CRM 사용자는 일부, 나머지는 엑셀+메모장+명함 서랍으로 영업
- → **이 사람들이 타겟**

## 9. 핵심 사용자 시나리오

### 시나리오 1: 운전 중 핸드폰 조회
1. 어제 노트북에서 거래처 "송재근" 정보 업데이트
2. 다음 날 운전 중 핸드폰에서 앱 열기
3. 오프라인 상태에서도 "송재근" 검색 → 정보 표시
4. 통화 후 새 메모 추가 → 온라인 복귀 시 자동 동기화

### 시나리오 2: 명함 스캔
1. 미팅 후 명함 받음
2. 카메라로 촬영
3. AI가 자동 파싱 → 이름/전화/회사/직급 추출
4. 확인 화면에서 수정 가능
5. 저장 → 거래처 자동 등록

### 시나리오 3: 엑셀 일괄 등록
1. 기존 엑셀로 관리하던 거래처 100명
2. 앱에 엑셀 업로드
3. 컬럼 매핑 (엑셀 컬럼 → 앱 필드)
4. 검증 후 일괄 등록 (성공/실패/건너뜀 표시)

### 시나리오 4: 월말 보고용 PDF
1. 이번 달 영업 건 필터링
2. PDF Export
3. 회사에 제출 (회사 양식이 아닌 본인 정리용)

### 시나리오 5: 마감 임박 알림
1. 마감 예정일 3일 전 자동 알림 (이메일 + 푸시 + **카카오톡 알림톡 예정**)
2. 당일 한 번 더 알림
3. 단계가 성사/실패면 알림 제외

### 시나리오 6: AI 회의록 작성
1. 거래처 미팅 시작 시 앱에서 "녹음 시작" 클릭
2. 1시간 미팅 후 "정지"
3. AI가 자동으로 회의록 요약 (날짜·회사·담당자·품목·진행단계·상세·향후계획·필요액션)
4. 사용자 확인/수정 후 저장
5. 해당 Deal과 연동 → Deal Activity 타임라인에 회의록 링크 자동 추가

## 10. 인증 / 알림 방향성

### 10.1 카카오 로그인만 사용 (1~3단계 웹)

**왜 카카오만**:
- 한국 영업사원이 1차 타겟 — 카카오 계정 보유율 거의 100%
- 회원가입 폼 없음 → 마찰 0
- 카카오는 검증된 이메일 자동 제공 → 별도 이메일 인증 불필요
- **카카오톡 알림 기능 추가 예정** — 같은 카카오 계정으로 연결되어 있어야 알림톡 발송 자연스러움

### 10.2 카카오톡 알림톡 (예정)

이메일/푸시 알림에 더해 **카카오톡 알림톡** 발송 검토:
- 마감 임박 딜 알림
- AI 회의록 생성 완료 알림
- 결제 성공 / 실패 알림
- 푸시보다 도달률 높음 (한국 사용자 대상)

### 10.3 iOS 출시 시 Apple 로그인 추가 (4단계)

Apple 앱스토어 정책상 소셜 로그인 제공 시 Apple Sign-in 강제 가능성 있음. 모바일 출시 시점에 추가 검토.
