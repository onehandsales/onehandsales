# Deal User Flow

## 1. 화면 목적

딜 목록 페이지는 사용자가 진행 중인 영업 기회를 빠르게 훑고, 같은 화면에서 선택한 딜의 상세와 다음 행동을 바로 확인하도록 만든다.

Desktop에서는 목록과 상세를 나란히 보여주는 split view를 기본으로 한다. Mobile에서는 목록을 먼저 보여주고, 항목 선택 시 상세 화면으로 전환한다.

## 2. 진입 조건

- 사용자는 로그인 상태여야 한다.
- User Web은 User API인 `/api/*`만 호출한다.
- Admin API인 `/admin/api/*`는 사용하지 않는다.
- 모든 데이터는 Backend에서 access token의 `userId` ownership으로 제한한다.

## 3. 주요 사용자 흐름

### 3.1 딜 목록 조회

1. 사용자가 딜 메뉴에 진입한다.
2. Frontend는 단계별 개수와 딜 목록을 조회한다.
3. 목록은 기본 10개씩 `createdAt DESC`로 표시한다.
4. 각 row에는 딜 이름, 회사, 담당자, 딜 단계, 금액, 마감일, 최신 다음 행동, 등록일, 최근수정일을 표시한다.
5. 회사, 담당자, 다음 행동은 객체 응답을 받아 UI에서 필요한 label로 조합한다.

### 3.2 검색, 필터, 정렬

1. 검색은 딜 이름만 대상으로 한다.
2. 필터는 회사, 담당자, 딜 상태를 대상으로 한다.
3. stage count는 검색, 회사, 담당자 필터를 반영한다.
4. 정렬은 기본 최신순이며 금액 높은 순, 금액 낮은 순, 마감일 빠른 순을 지원한다.
5. 검색/필터/정렬 변경 시 page는 1로 초기화한다.

### 3.3 상세 확인

1. 사용자가 목록 item을 선택한다.
2. Frontend는 딜 단건 상세를 조회한다.
3. 상세에는 딜 이름, 딜 금액, 회사, 담당자, 제품 목록, 마감일, 등록일, 최근수정일, 딜 상태를 표시한다.
4. 다음 행동 로그와 메모 로그는 상세 영역에서 별도 API로 조회한다.

### 3.4 딜 생성

1. 사용자가 딜 생성 버튼을 누른다.
2. Frontend는 회사, 담당자, 제품 옵션을 조회한다.
3. 사용자는 딜 이름, 딜 금액, 회사, 담당자, 제품 1개 이상, 딜 상태, 다음 행동, 예상마감일을 입력한다.
4. Backend는 담당자가 선택한 회사에 속하는지 검증한다.
5. Backend는 Deal을 생성하고 같은 transaction 안에서 DealProduct 목록과 최초 DealFollowingActionLog를 생성한다.
6. 최초 다음 행동 로그의 `checkComplete`은 `false`다.
7. 생성 후 목록, 단계별 개수, 상세 캐시를 갱신한다.

### 3.5 딜 수정

1. 사용자가 상세에서 수정을 시작한다.
2. 수정 가능한 값은 딜 이름, 딜 금액, 회사, 담당자, 제품 목록, 마감일, 딜 상태다.
3. 담당자 변경은 담당자 이름과 담당자 부서 표시가 함께 변경되는 결과를 만든다.
4. 제품 목록 변경은 `productIds` 배열을 보내 기존 딜-제품 연결을 교체한다.
5. Backend는 최종 담당자가 최종 회사에 속하는지 검증한다.
6. 수정 후 상세와 목록 캐시를 갱신한다.

### 3.6 다음 행동 로그

1. 상세 영역에서 다음 행동 로그 목록을 등록일 DESC로 조회한다.
2. 사용자는 `followingAction`만 입력해 새 다음 행동을 추가할 수 있다.
3. 새 로그는 `checkComplete=false`로 생성된다.
4. 사용자는 기존 로그의 다음 행동 문구와 완료 여부를 수정할 수 있다.
5. 딜 목록의 `latestFollowingAction`은 가장 최근에 생성된 로그 1개다.

### 3.7 메모 로그

1. 상세 영역에서 메모 로그 목록을 등록일 DESC로 조회한다.
2. 사용자는 `memoType`, `memo`를 입력해 새 메모를 추가할 수 있다.
3. 사용자는 기존 메모의 `memoType`, `memo`를 수정할 수 있다.

### 3.8 Export

1. 사용자가 목록에서 export를 실행한다.
2. Frontend는 현재 검색어, 회사 필터, 담당자 필터, 상태 필터, 정렬을 export API로 전달한다.
3. page는 전달하지 않는다.
4. Backend는 현재 조건에 맞는 전체 데이터를 xlsx로 반환한다.
5. export에는 id, 제품, 최근수정일이 포함되지 않는다.
