# MVP 유저 플로우

## 1. 목적

이 문서는 MVP 구현 시 Frontend와 Backend가 같은 사용자 흐름을 기준으로 작업하도록 정리한 실행용 유저 플로우다.

화면만 나열하지 않고, 사용자의 의도, 필요한 화면, Backend 처리, 저장되는 데이터까지 함께 설명한다.

## 2. 공통 전제

- 사용자는 소셜 로그인을 통해 가입하고 로그인한다.
- 모든 사용자 소유 데이터는 `userId`로 분리한다.
- 삭제는 즉시 완전 삭제가 아니라 30일 휴지통 보관이다.
- 민감 데이터는 기본적으로 보호하며, Admin 원문 조회 시 사유 입력과 감사 로그가 필수다.
- User Web은 `/api/*`만 호출한다.
- Admin Web은 `/admin/api/*`만 호출한다.

## 3. Flow 0. 로그인 후 딜 파이프라인 확인

### 사용자 의도

영업자는 앱을 열자마자 현재 진행 중인 딜, 금액, 가능성, 다음 행동, 지연 여부를 확인하고 싶다.

### 화면 흐름

1. 사용자가 `/login`에서 소셜 로그인을 선택한다.
2. 로그인 성공 후 `/` 딜 파이프라인 홈으로 이동한다.
3. 홈 상단에서 통합검색, 빠른 생성, 알림을 확인한다.
4. 중앙에서 단계 탭을 선택한다.
5. 딜 리스트에서 딜명, 회사/담당자, 단계, 금액, 가능성, 다음 행동, 마감일을 확인한다.
6. 데스크톱에서는 딜 선택 시 우측 상세 패널이 열린다.
7. 모바일에서는 딜 선택 시 상세 화면으로 이동한다.

### Backend 처리

- 현재 사용자 정보 조회
- 진행 중 딜 목록 조회
- 단계별 개수와 금액 요약 조회
- 오늘 일정과 임박/지연 다음 행동 조회
- 최근 활동 로그와 회의록 일부 조회

### 주요 데이터

- `User`
- `Deal`
- `DealActivity`
- `Company`
- `Contact`
- `ProductConnection`
- `Schedule`
- `MeetingNote`
- `Notification`

### 완료 기준

- 로그인 후 첫 화면에서 딜 파이프라인이 가장 먼저 보인다.
- 딜 리스트에는 비교에 필요한 핵심 필드가 모두 표시된다.
- 지연 또는 임박 다음 행동이 색상으로 구분된다.

## 4. Flow 1. 회사/거래처(담당자)/제품 기본 등록

### 사용자 의도

영업자는 딜을 만들기 전에 회사, 담당자, 제품 정보를 빠르게 등록하고 이후 상세 정보를 보강하고 싶다.

### 화면 흐름

1. 사용자가 상단 빠른 생성 버튼을 누른다.
2. 회사, 거래처(담당자), 제품 중 하나를 선택한다.
3. 빠른 등록 모달에서 최소 필드만 입력한다.
4. 저장 후 현재 화면에 남거나 상세 페이지로 이동한다.
5. 상세 페이지에서 태그, 메모, 추가 정보를 보강한다.

### Backend 처리

- 회사 생성, 수정, 목록, 상세, 삭제, 복구
- 거래처(담당자) 생성, 수정, 목록, 상세, 삭제, 복구
- 제품 생성, 수정, 목록, 상세, 삭제, 복구
- 태그 생성과 대상 연결
- 사용자 소유 데이터 필터링

### 주요 데이터

- `Company`
- `CompanyLog`
- `Contact`
- `Product`
- `ProductConnection`
- `Tag`
- `TagAssignment`

### 완료 기준

- 최소 정보만으로 회사/거래처(담당자)/제품을 빠르게 저장할 수 있다.
- 목록 검색과 상세 조회가 가능하다.
- 삭제한 데이터는 휴지통에서 복구할 수 있다.

## 5. Flow 2. 명함 OCR

### 사용자 의도

영업자는 명함 이미지를 업로드해서 회사와 담당자 정보를 자동 추출하고, 직접 확인한 뒤 저장하고 싶다.

### 화면 흐름

1. 사용자가 `/contacts/scan` 또는 `/business-cards`에서 명함 이미지를 업로드한다.
2. OCR 처리 중 상태를 확인한다.
3. AI가 추출한 회사명, 이름, 전화번호, 이메일, 부서, 직급을 확인한다.
4. 기존 회사 후보가 있으면 선택한다.
5. 기존 회사 연결, 새 회사 생성, 회사 없이 저장 중 하나를 선택한다.
6. 거래처(담당자)를 저장한다.

### Backend 처리

- 이미지 업로드 파일 검증
- Supabase Storage 또는 선택한 저장소에 이미지 저장
- OpenAI OCR port 호출
- OCR 결과 정규화
- 기존 회사 후보 검색
- 사용자가 확정한 결과로 `Company`, `Contact`, `BusinessCardScan` 저장

### 주요 데이터

- `BusinessCardScan`
- `Company`
- `Contact`
- `AuditLog`

### 완료 기준

- OCR 결과는 자동 저장되지 않는다.
- 사용자가 확인/수정 후 저장한다.
- 기존 회사 후보를 먼저 보여준다.

## 6. Flow 3. 딜 생성과 관리

### 사용자 의도

영업자는 회사/담당자/제품을 연결해 딜을 만들고, 단계와 활동 이력을 계속 관리하고 싶다.

### 화면 흐름

1. 사용자가 홈 또는 딜 목록에서 새 딜을 누른다.
2. 딜명, 회사, 거래처(담당자), 제품, 금액, 단계, 가능성을 입력한다.
3. 회사/거래처/제품이 없으면 모달 안에서 최소 정보로 즉시 생성한다.
4. 딜 저장 후 목록에 반영된다.
5. 딜 상세에서 단계, 가능성, 다음 행동, 메모를 수정한다.
6. 단계 변경 시 활동 로그가 자동 생성된다.
7. 사용자는 전화, 미팅, 이메일, 메모 등 활동 로그를 직접 추가한다.

### Backend 처리

- 딜 생성
- 제품 연결 생성
- 단계 변경 시 자동 활동 로그 생성
- 다음 행동 상태 계산 또는 저장
- 딜 목록 필터링
- 딜 상세 조회
- 딜 삭제와 복구

### 주요 데이터

- `Deal`
- `ProductConnection`
- `DealActivity`
- `DealActivityType`
- `Company`
- `Contact`
- `Schedule`
- `MeetingNote`

### 완료 기준

- 딜 금액은 필수다.
- 가능성 기본 UI는 긍정/중립/부정이다.
- 단계 변경은 반드시 활동 로그로 남는다.
- 다음 행동은 딜 목록과 상세에서 바로 보인다.

## 7. Flow 4. 일정 관리와 주간 일정 보고서

### 사용자 의도

영업자는 영업 일정을 딜/회사/담당자와 연결하고, 주간 보고서로 출력하고 싶다.

### 화면 흐름

1. 사용자가 일정 화면에서 새 일정을 만든다.
2. 일정 제목, 시작/종료 시간, 장소, 메모를 입력한다.
3. 필요하면 딜/회사/거래처(담당자)를 연결한다.
4. 알림 기본값을 확인하거나 수정한다.
5. 주간 일정표 화면에서 한 주 일정을 확인한다.
6. PDF 또는 Excel로 다운로드한다.

### Backend 처리

- 일정 CRUD
- 일정과 딜/회사/거래처 연결
- 일정 알림 생성
- 주간 일정 조회
- PDF/Excel ExportJob 생성

### 주요 데이터

- `Schedule`
- `ScheduleReminder`
- `Notification`
- `ExportJob`
- `Deal`
- `Company`
- `Contact`

### 완료 기준

- 일정은 딜 없이도 저장할 수 있다.
- 딜에서 만든 일정은 딜의 회사/거래처 정보를 기본 상속한다.
- 주간 보고서는 화면, PDF, Excel을 지원한다.

## 8. Flow 5. 구글 캘린더 가져오기

### 사용자 의도

영업자는 기존 구글 캘린더 일정을 서비스 안으로 가져와 딜/회사/담당자와 연결하고 싶다.

### 화면 흐름

1. 사용자가 설정 또는 일정 화면에서 Google Calendar 연결을 시작한다.
2. Google OAuth를 완료한다.
3. 가져올 기간 또는 캘린더를 선택한다.
4. 가져온 일정을 확인한다.
5. 필요하면 각 일정을 딜/회사/담당자와 연결한다.

### Backend 처리

- Google OAuth token 저장
- Google Calendar event 조회
- 중복 event 확인
- `Schedule.source = GOOGLE`로 저장
- 외부 event id 저장

### 주요 데이터

- `ExternalCalendarConnection`
- `Schedule`

### 완료 기준

- MVP에서는 가져오기만 한다.
- 우리 서비스 일정을 Google Calendar로 내보내지 않는다.
- 양방향 동기화는 하지 않는다.

## 9. Flow 6. AI 회의록

### 사용자 의도

영업자는 회의 내용을 대략 입력하고, AI가 정리한 회의록을 수정해 저장한 뒤 딜 활동 이력과 연결하고 싶다.

### 화면 흐름

1. 사용자가 `/meetings/new`에서 회의 내용을 텍스트로 입력한다.
2. AI 회의록 생성을 실행한다.
3. AI가 날짜, 회사, 담당자, 부서, 품목, 진행단계, 상세내용, 향후계획, 필요액션을 생성한다.
4. 사용자가 결과를 수정한다.
5. 회사/담당자/딜 후보를 확인한다.
6. 회의록을 저장한다.
7. 필요하면 딜과 연결한다.
8. 딜 연결 시 딜 활동 로그가 자동 생성된다.

### Backend 처리

- OpenAI meeting-note port 호출
- AI 결과 schema 검증
- 후보 회사/담당자/딜 검색
- 회의록 저장
- 딜 연결 시 활동 로그 생성

### 주요 데이터

- `MeetingNote`
- `AiJob`
- `Deal`
- `DealActivity`
- `Company`
- `Contact`

### 완료 기준

- 회의록은 딜 없이도 저장할 수 있다.
- 딜 연결 시 활동 로그가 자동 생성된다.
- AI 결과는 사용자가 수정한 뒤 저장한다.

## 10. Flow 7. Import

### 사용자 의도

영업자는 기존 Excel/CSV 데이터를 업로드해 회사, 거래처(담당자), 제품, 딜 데이터를 빠르게 가져오고 싶다.

### 화면 흐름

1. 사용자가 `/imports`에서 대상 유형을 선택한다.
2. Excel 또는 CSV 파일을 업로드한다.
3. AI가 컬럼 자동 매핑 결과를 제안한다.
4. 사용자가 매핑을 확인하고 수정한다.
5. Import를 확정한다.
6. 성공, 실패, 건너뜀 결과를 확인한다.

### Backend 처리

- 파일 업로드 검증
- Excel/CSV 파싱
- OpenAI import-mapping port 호출
- 컬럼 매핑 저장
- 사용자 확정 후 batch import 실행
- row별 성공/실패 결과 저장

### 주요 데이터

- `ImportJob`
- `ImportJobRow`
- `AiJob`
- `Company`
- `Contact`
- `Product`
- `Deal`

### 완료 기준

- AI 매핑 결과는 사용자가 확인한 뒤 확정한다.
- 실패 row를 확인할 수 있다.
- 일정/회의록 Import는 MVP에서 제외한다.

## 11. Flow 8. Export

### 사용자 의도

영업자는 회사, 거래처, 제품, 딜, 일정, 회의록 데이터를 PDF 또는 Excel로 내려받고 싶다.

### 화면 흐름

1. 사용자가 `/exports`에서 Export 대상을 선택한다.
2. PDF 또는 Excel을 선택한다.
3. 민감 데이터 포함 여부를 선택한다.
4. 민감 데이터 포함 시 경고를 확인한다.
5. 파일 생성 후 다운로드한다.

### Backend 처리

- Export 대상 검증
- 민감 데이터 포함 여부 저장
- 경고 확인 여부 저장
- 파일 생성 job 실행
- 생성 파일 다운로드 URL 제공

### 주요 데이터

- `ExportJob`
- `AuditLog`

### 완료 기준

- 민감 데이터는 기본 제외다.
- 사용자가 명시적으로 포함할 수 있다.
- 민감 데이터 포함 시 경고가 반드시 표시된다.

## 12. Flow 9. 통합검색

### 사용자 의도

영업자는 기억나는 단서 하나로 회사, 담당자, 제품, 딜, 일정, 회의록을 빠르게 찾고 싶다.

### 화면 흐름

1. 사용자가 상단 통합검색에 키워드를 입력한다.
2. 결과가 회사, 거래처, 제품, 딜, 일정, 회의록으로 묶여 표시된다.
3. 진행 중 딜과 최근 항목이 우선 표시된다.
4. 결과를 선택하면 상세 페이지 또는 상세 패널로 이동한다.

### Backend 처리

- 사용자 소유 데이터 범위에서 통합검색
- entity type별 결과 grouping
- 활성 딜과 최근 항목 우선 정렬
- 민감 데이터 본문은 검색 결과에서 과도하게 노출하지 않음

### 주요 데이터

- `Company`
- `Contact`
- `Product`
- `Deal`
- `Schedule`
- `MeetingNote`

### 완료 기준

- 통합검색은 전체 탐색용이다.
- 화면별 검색/필터는 현재 목록 제어용으로 별도 유지한다.

## 13. Flow 10. Admin 민감정보 조회

### 사용자 의도

Admin은 운영상 필요한 경우 전체 데이터를 확인할 수 있지만, 민감정보 원문은 통제된 절차로만 조회해야 한다.

### 화면 흐름

1. Admin이 `/admin/login`에서 로그인한다.
2. 사용자 목록 또는 전체 딜/회사/거래처/제품 목록에 진입한다.
3. 민감 데이터는 기본 마스킹 상태로 표시된다.
4. 원문 보기가 필요한 경우 명시적 버튼을 누른다.
5. 사유 입력 dialog가 열린다.
6. 사유를 입력하고 확인하면 원문 데이터가 표시된다.
7. 감사 로그 화면에서 해당 조회 기록을 확인할 수 있다.

### Backend 처리

- AdminGuard 확인
- raw sensitive view reason 검증
- 원문 조회와 감사 로그 기록을 같은 transaction으로 처리
- Admin response에 필요한 범위만 원문 포함

### 주요 데이터

- `User`
- `AuditLog`
- `Deal`
- `MeetingNote`
- `PersonalMemo`

### 완료 기준

- Admin 목록은 민감 데이터를 기본 마스킹한다.
- 원문 조회는 사유 없이는 실패한다.
- 원문 조회는 반드시 감사 로그를 남긴다.
