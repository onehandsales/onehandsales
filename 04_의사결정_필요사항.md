- 온라인 복귀 시 큐 순차 처리

- 명함 이미지는 임시 저장 → 온라인 복귀 시 업로드 + OCR 호출

**적용 대상**: Company / Contact / Deal / DealActivity / Product / Note 모든 도메인

**구현 위치**: 모바일 3단계 진입 시 [03_기능_및_UserFlow.md](03_기능_및_UserFlow.md) Flow K 갱신.

### D-07. OCR 결과 신뢰도 낮을 때 UX 🟡

**무엇을 결정해야 하는가**: 명함을 찍었는데 GPT Vision이 "이름 = 김??" 처럼 흐릿하게 읽었을 때 어떻게 보여줄지.

**왜 결정이 필요한가**:
- AI OCR은 100% 정확하지 않음. 흑백 명함, 손글씨 메모, 영문/일본어, 사진 흐림 등 다양한 실패 케이스가 있음
- 그대로 저장하면 "김??"가 거래처로 등록 → 검색 망함
- 사용자가 "이 앱 이상해" 결론 내릴 수 있음

**선택지**:
- (A) **신뢰도 임계값 설정** — 0.7 이상만 자동 저장, 미만은 수동 입력 화면으로 전환
- (B) **무조건 사용자 확인 화면 거치기** — 신뢰도 무관하게 매번 확인 (안전, 사용자 마찰 ↑)
- (C) **실패 시 이미지만 저장 + "OCR 대기" 상태** — 나중에 재시도 가능

**관련 사항**:
- 영문/일본어 명함 처리 (GPT Vision은 다국어 OK)
- 실패 명함을 모아 재시도 큐 → GPT Vision 모델 업데이트 시 일괄 재처리

**왜 결정이 필요한가**: "Admin 대시보드" 만 합의됨. 구체 지표/액션이 안 정해져 있음.

**결정 항목**:

- **감사 로그 보관 기간**: 1년? 3년? 영구? (법적 요구 vs Storage 비용)

**3단계(웹 외부 공개)에는 결정 필요**. 그 전엔 화면 골격만 있어도 OK.

## 3. 신규 결정 필요 🟢

### D-09. 거래처 합치기 (병합) — 되돌리기 불가 정책 🟢

**무엇을 결정해야 하는가**: "김철수"가 실수로 2번 등록됐을 때 합치기 기능. 합친 후 되돌릴 수 있는가?

**왜 결정이 필요한가**: 새 기획은 "되돌릴 수 없음 + 확인 다이얼로그" 명시. 하지만 실수 시 위험.

**선택지**:
- (A) **새 기획대로 — 완전 비가역** — 단순. 사용자가 다이얼로그에서 한 번 더 확인. 잘못해도 사용자 책임
- (B) **30일 휴지통처럼 복구 기간** — 안전망 제공. 구현 복잡 (병합 이력 별도 테이블 필요)
- (C) **병합 전 백업 자동 생성** — 사용자가 "병합 전 상태 복원" 버튼 클릭 가능. 백업 데이터 보관 비용

**제 의견**: (A) — 휴지통은 "삭제 실수" 안전망인데, 합치기는 "통합 의도"라 의도성이 강함. 다이얼로그만 강하게.

### D-10. Deal 단계 역방향 전이 허용 여부 🟢

**무엇을 결정해야 하는가**: 딜을 "성사"로 바꿨다가 잘못 눌렀을 때 "협의중"으로 되돌릴 수 있는가?

**단계 흐름**: 초기접촉 → 협의중 → 성사 OR 실패

**왜 결정이 필요한가**:
- 잘못 누르는 실수는 발생함
- 하지만 "단계 변경: 성사 → 협의중" 자동 활동 기록이 어색함 ("이 딜은 성사됐었는데 다시 협의 중?" → 데이터 신뢰도 ↓)

**선택지**:
- (A) **자유롭게 모든 방향 전이 허용** — 단순. 실수 정정 쉬움. 데이터 의미 약화
- (B) **앞으로만 전이 가능** — 한 번 성사/실패면 잠금. 잘못 누른 경우 "원복" 별도 기능 필요
- (C) **앞으로 자유 + 뒤로는 "원복" 버튼** — 단계 변경 직후 1시간 내 원복 가능. 그 외 잠금

**제 의견**: (A) — 개인용 도구라 본인이 알아서 관리. UI에서 단계 클릭 시 다이얼로그 한 번만.

### D-11. 엑셀 Import 컬럼 매핑 프리셋 저장 🟢

**무엇을 결정해야 하는가**: 엑셀 가져올 때 매번 "엑셀 컬럼 A = 앱의 이름 필드" 식으로 매핑하는데, 다음 번에 자동 적용할지.

**왜 결정이 필요한가**: 같은 사용자가 같은 엑셀 양식을 반복 사용함. 매번 매핑하면 짜증 → 이탈 위험.

**선택지**:
- (A) **MVP는 매번 매핑** — 단순 구현. 사용자 마찰
- (B) **프리셋 저장 + 자동 적용** — 첫 매핑 후 "프리셋 저장" 체크 → 다음에 자동 적용. UX 우위

**제 의견**: (B). 어차피 매핑 화면은 만들어야 하니 저장 한 줄 추가 정도. 구현 비용 작음.

### D-12. Deal 메모 검색 포함 vs 오프더레코드 분리 명확화 🟢

**무엇을 결정해야 하는가**: 검색창에 "예산 풀림" 쳤을 때 뭐가 나와야 하는가?

**현재 정책**:
- ✅ 검색 대상: 딜명 / 담당자명 / 회사명 / **딜의 일반 메모**
- ❌ 검색 제외: **오프더레코드 메모** (Company / Contact / Deal 어디에도)

**왜 결정이 필요한가**:
- "딜 메모"와 "오프더레코드 메모"의 차이가 UI에서 명확해야 사용자가 헷갈리지 않음
- 사용자는 "내가 적은 거 다 검색되는 줄 알았는데"라고 생각할 수 있음

**검토 사항**:
- UI에서 "공개 메모" vs "비공개 메모" 같은 라벨로 명확히 구분
- 오프더레코드 메모는 자물쇠 아이콘 등 시각적 구분
- Contact / Company / Deal 모두 동일 정책 (오프더레코드는 일관 제외)

### D-14. 데이터 영구 삭제 정책 (휴지통 30일 이후) 🟢

**무엇을 결정해야 하는가**: 휴지통의 데이터가 30일 후 완전 삭제될 때 사용자에게 어떻게 알릴지.

**왜 결정이 필요한가**:
- "어? 거래처가 사라졌어요" 사고 100% 발생
- 영업 데이터는 실수로 지워지면 매출 직결

**선택지**:
- (A) **30일 자동 삭제 + 사전 알림 X** — 단순. 사용자 책임
- (B) **삭제 7일 전 이메일/푸시 알림** — 친절. 구현 비용 약간
- (C) **삭제 직전 한 번 더 확인 알림** — 가장 안전. 사용자 마찰

**추가 검토**:
- Admin이 강제 복구 가능? (감사 로그 + 사유 입력)
- 결제 영수증/감사 로그 같은 법적 보관 데이터는 별도 정책 (Hard Delete 금지)

**제 의견**: (B). 7일 전 이메일 + 푸시. 사용자가 휴지통 확인하고 복구 가능.

### D-15. Product 단가 화폐 단위 🟢

**무엇을 결정해야 하는가**: 제품 단가 필드에 1000 입력했을 때 1,000원? 1,000달러?

**왜 결정이 필요한가**:
- 한국 영업사원이 1차 타겟 → KRW 기본
- 반도체 부품 영업처럼 해외 거래하는 영업사원도 있음

**선택지**:
- (A) **KRW 고정** — 단순. 1차 타겟에 적합. 해외 거래는 사용자가 환산해서 입력
- (B) **다중 통화 지원** — Product마다 통화 선택. Deal에도 통화 표시. 환율 변환 필요
- (C) **사용자 기본 통화 설정** — 설정에서 한 번 정하면 모든 데이터에 적용. 변경 시 마이그레이션

**제 의견**: 1차는 (A). 사용자 누적 후 (C). (B)는 환율 처리가 복잡해서 비추.

### D-16. 계정 삭제 / 데이터 내보내기 (법적 의무) 🟢

**무엇을 결정해야 하는가**: 회원 탈퇴 기능 + 본인 데이터 다운로드 기능.

**왜 결정이 필요한가**:
- 한국 개인정보보호법: 본인 데이터 열람·삭제권 의무
- Apple 앱스토어: 2022년부터 "계정 삭제 기능" 강제 (없으면 앱 거부)
- 사용자 신뢰 핵심: "내 데이터는 내가 가져갈 수 있다" 약속

**선택지**:
- 탈퇴 시점: **즉시 삭제 vs 7일 유예** (실수 방지 vs 단순)
- 데이터 다운로드 포맷: **엑셀** (사용자 친화) / **JSON** (완전성) / **둘 다**
- 명함 원본 이미지도 함께 내려줄지

**제 의견**:
- 탈퇴는 7일 유예 (실수 방지 + 환불 처리 시간)
- 데이터는 엑셀 + 명함 이미지 zip 묶음
- 결제/감사 로그는 법정 보관 기간 (5년) 별도 보관 (탈퇴와 무관)

---

### D-17. 비공식 인사이트 (오프더레코드) 데이터 보안 🟢

**무엇을 결정해야 하는가**: "삼성 김부장이 진짜 결정권자임" 같은 민감 메모를 어떻게 안전하게 저장할지.

**왜 결정이 필요한가**:
- 영업사원이 가장 비공식적이고 민감한 정보를 적는 공간
- DB 유출 시 영업 비밀 + 인맥 정보 노출 → 서비스 신뢰 추락
- Admin이 사용자 메모를 볼 수 있는가? 이게 윤리적 이슈

**선택지**:
- (A) **일반 데이터와 동일하게 저장** — 단순. RLS만으로 격리
- (B) **사용자별 암호화 키로 추가 암호화** — 사용자만 복호화 가능. 비밀번호 분실 시 데이터 완전 손실
- (C) **Admin 접근 차단 정책** — 기술적으로는 접근 가능하지만 정책상 금지 + 감사 로그
- (D) **클라이언트 사이드 암호화** — 백엔드는 암호문만 저장. 검색/동기화 어려움

**제 의견**: (A) + (C) 조합. 일반 데이터처럼 저장하되 Admin이 오프더레코드 메모 조회 시 감사 로그 강제 기록 + 정책상 금지.

## 4. 결제 / 구독 관련 신규 결정 🟢 (3단계 외부 공개 전 결정 필수)

### D-18. PG사 / 결제 수단 선택 🟢

**상황**: 카드 결제 처리에 어떤 PG사를 쓸지 미정. 계좌이체는 초기에 가능.

**선택지**:
- (A) **Toss Payments** — 한국 표준, 카카오페이 / 카드 / 계좌이체 통합. 수수료 ~3%
- (B) **KakaoPay 단독** — 카카오 로그인 사용자 흐름 자연스러움
- (C) **Stripe** — 글로벌 표준, 해외 진출 시 유리. 한국 가맹점 등록 복잡
- (D) **NicePay / KG이니시스** — 전통 PG사

**검토 사항**:
- 카드 토큰 보관 방식 (PG사 별 vault)
- 정기 결제 (subscription billing) 지원
- 환불 API 자동화 가능성

---

### D-19. Plan 등급 / 기능 제한 정책 🟢

**상황**: Free / Pro / Enterprise 등급 별 제한 미정.

**검토 사항**:
- Free 사용자 제한 (Contact 50개? Deal 20개? Storage 100MB?)
- Pro 가격대 (월 9,900원? 19,900원?)
- 트라이얼 기간 (7일? 14일? 30일?)
- 명함 OCR 사용량 제한 (GPT Vision 비용 직결)
- Enterprise는 도입 시점 미정 — 초기엔 Free + Pro만

---

### D-20. 무료 사용자 데이터 정책 🟢

**상황**: 트라이얼 종료 후 결제 안 하면 데이터는?

**선택지**:
- (A) **읽기 전용 모드 전환** — 데이터는 보존, 추가 입력 불가
- (B) **30일 유예 후 데이터 삭제 경고** — 결제 유도
- (C) **Free 등급으로 강제 다운그레이드** — 상한 초과분만 잠금

---

### D-21. 환불 자동화 범위 🟢

**상황**: 7일 이내는 자동 승인 (전자상거래법). 7일 이후는?

**선택지**:
- (A) **Admin 검토 후 처리** — 사례별 판단
- (B) **자동 거부 + 사용자 지원 채널 안내** — 단순
- (C) **부분 환불 자동화** — 사용 기간 비례 계산

---

### D-22. 영수증 / 세금계산서 발급 🟢

**상황**: 한국 법상 영수증 발급 의무 + 사업자 사용자는 세금계산서 요청 가능.

**검토 사항**:
- PDF 영수증 자동 생성 (PG사 또는 자체)
- 세금계산서는 어떻게? (홈택스 연동? 수동?)
- 사업자 등록번호 입력 필드
