## 1. 개발 전략

### 1.1 세로 슬라이스(Vertical Slice) 개발 원칙

기능 단위로 **DB → BE API → FE UI → 배포 → 동작 확인**까지 한 번에 끝내고 다음 기능으로.

**❌ 나쁜 순서 (절대 금지)**
```
DB 전부 만들기 → API 전부 만들기 → UI 전부 만들기
```
- 끝까지 가서야 동작 확인 가능
- 중간에 설계 오류 발견하면 다 갈아엎어야 함
- 사용자 피드백 받기까지 시간이 오래 걸림

**✅ 좋은 순서 (반드시 준수)**
```
로그인 하나 완성 → 배포 → 동작 확인
  → 거래처 등록 하나 완성 → 배포 → 동작 확인
  → 거래처 목록 조회 완성 → 배포 → 동작 확인
  → 검색 완성 → 배포 → 동작 확인
  → 엑셀 export 완성 → 배포 → 동작 확인
```

### 1.2 AI 협업 원칙
- **세로 슬라이스 책임 분담**: 한 명이 BE만, 한 명이 FE만 X → 각자 한 슬라이스를 끝까지 책임
- **API 계약 우선**: 새 기능은 `AGENT/api-contract.md`에 엔드포인트 먼저 적고 시작
- **컨텍스트 분리**: AI에게 BE 컨텍스트와 FE 컨텍스트를 분리해서 줌 (레포 분리의 효과)

### 1.3 테스트 정책 (테스트 코드 거의 없음)
- 세로 슬라이스 + 직접 클릭 QA로 검증
- 매주 회귀 시나리오 체크리스트 운영
- **예외 2가지**: 인증/권한 로직, 엑셀 Import 파싱 → 통합 테스트 3~5개

### 1.4 부가 기능 게이트키핑
- 견적서, 회의 녹음, 자체 캘린더는 만들지 않음
- 요청 와도 일단 `TODO/backlog.md`에만 기록
- 정체성 흔드는 기능은 거절

## 4. AI 협업 운영 룰

### 4.1 `AGENT/CLAUDE.md`에 박을 핵심 룰
1. **세로 슬라이스 원칙**: 한 기능을 시작했으면 DB → BE → FE → 배포 → QA까지 한 번에 끝낸다
2. **부가 기능 금지**: 견적서, 회의 녹음, 자체 캘린더 만들지 않는다
3. **API 계약 우선**: `AGENT/api-contract.md`에 엔드포인트 먼저 적고 시작
4. **FE/BE 동시 작업**: 각자 한 슬라이스를 끝까지 책임
5. **테스트 없음 정책**: 단 인증/엑셀 Import는 예외
6. **Railway 전용 기능 금지**: Cron, Volumes, Private Network 안 씀

### 4.2 작업 시작 시 AI에게 줄 정보
- `AGENT/CLAUDE.md` (전체 컨텍스트)
- 해당 기능의 `TODO/00X-feature.md` (이번 작업 명세)
- `AGENT/api-contract.md` 관련 부분
- `AGENT/conventions.md` (코딩 컨벤션)

### 4.3 AI 산출물 검증 방식
- 코드 작성 후 **반드시 본인이 한 번 읽고** 이상한 부분 수정
- 배포 후 PC + 모바일에서 직접 클릭 테스트
- Sentry 에러 모니터링 (배포 후 5분간)
