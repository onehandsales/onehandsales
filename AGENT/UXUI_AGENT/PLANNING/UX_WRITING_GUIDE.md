# UX Writing Guide

> 기준 출처: Toss UX Writing Guide (`https://developers-apps-in-toss.toss.im/design/ux-writing.md`)를 `한손에 영업 / onehand.sales` 제품 문맥에 맞게 적용한다.

## 1. 목적

제품 안의 모든 사용자 노출 문구는 쉽고 짧고 행동이 분명해야 한다.

`한손에 영업 / onehand.sales`는 개인 영업자가 반복해서 쓰는 업무 도구이므로, 문구는 친근하되 가볍지 않고, 영업 업무를 빠르게 이해하고 다음 행동으로 이어지게 해야 한다.

## 2. 적용 범위

우선 적용 범위:

- User Web의 화면 텍스트
- 버튼, 탭, 메뉴, 필터, placeholder
- toast, dialog, modal
- loading, empty, error state
- form validation message
- API error message 중 사용자에게 노출되는 문구

Admin Web도 사용자에게 보이는 제품 문구에는 같은 원칙을 적용한다. 다만 Admin은 운영 콘솔이므로 표 제목, 테이블 헤더, 필터명처럼 업무 용어가 더 정확한 곳은 짧은 명사형을 허용한다.

## 3. 기본 원칙

### 해요체를 쓴다

제품 안의 문장은 상황과 맥락을 불문하고 해요체를 기본으로 쓴다.

피해야 할 표현:

- `~입니다`
- `~습니다`
- `~합니다`
- `~없습니다`
- `~하지 않습니다`
- `~수 없습니다`
- `~못했습니다`
- `~필요합니다`
- `~되었습니다`

권장 표현:

- `~이에요`
- `~해요`
- `~했어요`
- `~하고 있어요`
- `~할 수 있어요`
- `~해 보세요`

예:

| 피하기 | 권장 |
|---|---|
| 불러오는 중입니다. | 불러오고 있어요. |
| 저장되었습니다. | 저장했어요. |
| 등록된 회사가 없습니다. | 새 회사를 등록하면 여기에서 볼 수 있어요. |
| 요청을 처리할 수 없습니다. | 잠시 후 다시 시도해 주세요. |
| 로그인이 필요합니다. | 로그인하면 이용할 수 있어요. |

### 능동적으로 말한다

사용자가 한 행동은 시스템이 수동적으로 처리한 것처럼 말하지 않는다.

예:

| 피하기 | 권장 |
|---|---|
| 회사가 등록되었습니다. | 회사를 등록했어요. |
| 딜이 삭제되었습니다. | 딜을 삭제했어요. |
| 일정이 복구되었습니다. | 일정을 복구했어요. |
| 파일이 업로드되었습니다. | 파일을 올렸어요. |

### 부정형보다 긍정형을 먼저 쓴다

`없어요`, `안 돼요`, `할 수 없어요`를 바로 쓰기보다 사용자가 무엇을 하면 되는지 알려준다.

예:

| 피하기 | 권장 |
|---|---|
| 검색 결과가 없습니다. | 검색어를 바꾸면 더 많은 결과를 찾을 수 있어요. |
| 연결된 딜이 없습니다. | 딜을 연결하면 이곳에서 함께 볼 수 있어요. |
| 오류 행이 있어 확정할 수 없습니다. | 오류 행을 수정하면 가져오기를 확정할 수 있어요. |
| 회사를 선택하지 않으면 담당자를 선택할 수 없습니다. | 회사를 선택하면 담당자를 고를 수 있어요. |

단, 위험 액션, 보안, 권한, 결제, 데이터 삭제처럼 오해가 더 위험한 상황에서는 문제를 명확히 말하고 바로 다음 행동을 붙인다.

예:

- 권한이 만료됐어요. 다시 로그인해 주세요.
- 이 작업은 되돌릴 수 없어요. 삭제하려면 한 번 더 확인해 주세요.

### 과도한 경어를 줄인다

캐주얼한 경어를 쓴다. `~시겠어요?`, `~시나요?`, `~께`, `계시다`, `여쭈다`처럼 과한 높임은 피한다.

예:

| 피하기 | 권장 |
|---|---|
| 저장하시겠어요? | 저장할까요? |
| 담당자가 계신가요? | 담당자가 있나요? |
| 고객님께 알려드려요. | 고객에게 알려요. |
| 정보를 여쭤볼게요. | 정보를 확인할게요. |

질문에서 `~시`를 빼면 어색할 때는 사용자를 주어로 만들지 말고 확인하려는 정보를 주어로 쓴다.

예:

- `어떤 회사를 담당하시나요?`보다 `담당 회사가 어디인가요?`

### 한 줄로 말한다

상태 문구와 안내 문구는 가능한 한 한 줄로 쓴다.

원칙:

- 한 문장에 하나의 의미만 담는다.
- 에러는 원인보다 해결 행동을 먼저 보이게 한다.
- 긴 설명은 본문이 아니라 보조 텍스트, tooltip, detail 영역으로 분리한다.

예:

| 피하기 | 권장 |
|---|---|
| CSV, XLS, XLSX 형식의 10MB 이하 파일을 지원합니다. | 10MB 이하 CSV, XLS, XLSX 파일을 올릴 수 있어요. |
| 행별 데이터와 오류 사유를 확인합니다. | 행별 오류를 확인해요. |

### `{명사} + {명사}`를 줄인다

한자어 명사나 딱딱한 업무 명사를 연속해서 쓰지 말고, 가능한 한 동사 형태로 풀어쓴다.

예:

| 피하기 | 권장 |
|---|---|
| 파일 업로드 | 파일 올리기 |
| 데이터 확정 | 데이터 확정하기 |
| 오류 확인 | 오류 확인하기 |
| 딜 복구 완료 | 딜을 복구했어요. |

다만 메뉴, 테이블 헤더, 필터, 고정 도메인 용어는 짧은 명사형을 허용한다.

허용 예:

- 딜
- 회사
- 담당자
- 제품
- 일정
- 회의록
- 명함 스캔
- 휴지통
- 엑셀 다운로드

## 4. 상황별 작성 규칙

### Empty state

빈 상태는 `없습니다`로 끝내지 않는다. 사용자가 다음에 할 수 있는 일을 보여준다.

예:

| 피하기 | 권장 |
|---|---|
| 등록된 회사가 없습니다. | 새 회사를 등록하면 여기에서 볼 수 있어요. |
| 연결된 담당자가 없습니다. | 담당자를 연결하면 이곳에서 함께 볼 수 있어요. |
| 이번 주 일정이 없습니다. | 일정을 등록하면 이번 주 계획을 한눈에 볼 수 있어요. |

### Loading state

`~중입니다` 대신 `~하고 있어요`를 쓴다.

예:

| 피하기 | 권장 |
|---|---|
| 검색 중입니다. | 검색하고 있어요. |
| 딜을 불러오는 중입니다. | 딜을 불러오고 있어요. |
| 세션을 확인하는 중입니다. | 세션을 확인하고 있어요. |

### Success toast

수동형 `~되었습니다` 대신 사용자의 행동을 능동형으로 말한다.

예:

| 피하기 | 권장 |
|---|---|
| 저장되었습니다. | 저장했어요. |
| 추가되었습니다. | 추가했어요. |
| 삭제되었습니다. | 삭제했어요. |
| 복구되었습니다. | 복구했어요. |

### Error state

문제가 생겼다는 사실은 짧게 말하고, 사용자가 할 수 있는 행동을 바로 붙인다.

예:

| 피하기 | 권장 |
|---|---|
| 요청을 처리하지 못했습니다. | 문제가 생겼어요. 잠시 후 다시 시도해 주세요. |
| 목록을 불러오지 못했습니다. | 목록을 불러오지 못했어요. 다시 시도해 주세요. |
| 삭제할 수 없습니다. | 연결된 데이터를 정리하면 삭제할 수 있어요. |

### Validation

입력 검증 문구는 짧고 구체적으로 쓴다. 단순 필수 입력은 `입력해 주세요`를 쓸 수 있지만, 가능한 경우 조건과 이유를 함께 말한다.

예:

| 피하기 | 권장 |
|---|---|
| 회사명을 입력해주세요. | 회사명을 입력해 주세요. |
| 금액은 0 이상의 정수입니다. | 금액은 0원 이상으로 입력해 주세요. |
| 날짜 형식은 YYYY-MM-DD입니다. | 날짜는 YYYY-MM-DD 형식으로 입력해 주세요. |
| 이미지 용량은 10MB 이하여야 합니다. | 10MB 이하 이미지만 올릴 수 있어요. |

### Dialog와 button

버튼은 짧고 행동 중심으로 쓴다.

원칙:

- 주요 행동: `저장`, `등록`, `삭제`, `복구`, `다운로드`
- 보조 행동: `닫기`
- 다이얼로그 왼쪽 버튼은 `닫기`로 통일한다.
- `취소`는 사용자가 진행 중인 작업 자체가 취소된다고 오해할 수 있으므로 신중하게 쓴다.
- 버튼에는 문장형 안내를 넣지 않는다.

예:

| 피하기 | 권장 |
|---|---|
| 저장하시겠습니까? | 저장할까요? |
| 취소 | 닫기 |
| 파일을 다운로드합니다 | 다운로드 |

## 5. 코드 리뷰 검색어

UX writing 검수 시 아래 표현을 우선 검색한다.

```text
입니다
습니다
합니다
없습니다
않습니다
수 없습니다
못했습니다
필요합니다
되었습니다
해주세요
시겠어요
시나요
께
계시
여쭈
```

`해주세요`는 해요체이지만, 붙여 쓰지 말고 `해 주세요`로 쓴다. 또한 모든 상황을 요청형으로 끝내지 말고, empty/error에서는 사용자가 얻는 결과나 다음 행동을 함께 보여준다.

## 6. 적용 우선순위

1. User Web에서 실제 사용자에게 보이는 문구
2. 공용 컴포넌트의 loading, empty, error state
3. form schema validation message
4. toast/dialog/modal 문구
5. API client의 사용자 노출 error fallback
6. Admin Web 운영자 노출 문구
