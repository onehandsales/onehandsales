## 1. 제품 정체성

- 서비스는 **B2C 개인 영업 전용 도구**로 시작한다.
- 주 타겟은 **남성 30~50대 영업직**, 특히 40~50대 가능성이 높다.
- 제품 정체성은 하나로 좁히지 않고 다음을 모두 포함한다.
  - 개인용 경량 CRM
  - 현장 영업 카드 앱
  - 딜/파이프라인 관리 도구
  - 일정/회의록 중심 영업 업무 관리 도구
- 초기 가격 가설은 월 구독 **5,900~6,900원**이다.

## 2. 서비스명

- 한국명: **한손에 영업**
- 영문명/도메인 후보: **onehand.sales**

## 3. MVP 핵심 루프

MVP는 아래 네 가지 루프를 모두 포함한다.

- 담당자/회사/제품 등록 및 빠른 검색
- 딜 생성, 단계/가능성/금액/활동 로그/Memo 기록 관리
- 일정 등록, 월간/주간 보기, 주간 일정 보고서 출력
- 회의 내용 작성, AI 회의록 생성, 딜 연결

개발 우선순위:

1. 회사/담당자/제품
2. 딜
3. 일정
4. 회의록

## 4. 플랫폼 전략

- **웹 MVP를 먼저 만든다.**
- 핵심 검증 후 iOS/Android 앱으로 확장한다.
- 모바일 앱은 React Native + Expo로 만든다.

## 5. 인증

MVP 인증 제공자는 다음 네 가지다.

- 카카오
- 구글
- 네이버
- 애플

애플 로그인은 iOS 앱 출시와 추후 모바일 구독 결제를 고려해 필수로 포함한다.

## 6. 결제/구독

- 결제/구독은 MVP에서 제외한다.
- MVP 이후 초기 유료 운영은 **계좌이체 수동 처리**로 시작한다.
- 약 1~2개월 동안 유지보수와 고객 니즈 반영을 진행한다.
- 이후 웹과 앱 모두에 **구독형 결제**를 붙인다.
- 단건 결제는 하지 않는다.

## 7. 영업 분야

- MVP는 특정 1개 영업 분야에 고정하지 않는다.
- 여러 영업 분야가 쓸 수 있는 **범용 구조**로 만든다.
- 이후 고객 반응을 보고 특정 분야별 템플릿/기능으로 확장한다.

## 8. 데이터 모델 방향

## 9. 회사와 담당자

- `담당자`는 회사가 아니라 **회사에 소속된 사람**, 즉 Contact를 의미한다.
- 구조는 `Company -> Contact`다.
- 화면 메뉴명은 **담당자** 로 사용한다.
- 회사 위치를 기본값으로 사용한다.
- 담당자 위치는 선택 입력이다.
- 회사의 `분야`는 `CompanyField` 테이블로 관리하고 회사는 `companyFieldId` FK를 가진다.
- 회사의 `지역`은 `CompanyRegion` 테이블로 관리하고 회사는 `companyRegionId` FK를 가진다.
- 회사 분야와 회사 지역은 사용자가 생성/삭제할 수 있지만, 이미 회사에 매핑된 값은 삭제할 수 없다.
- 회사 분야와 회사 지역 수정 API는 현재 제공하지 않는다.
- 회사명, 회사분야, 회사지역은 회사 단건 수정 API로 변경할 수 있다.

## 10. 회사 메모와 개인 비밀 메모

- 회사 특징 메모는 `CompanyMemoLog` 테이블에 기록한다.
- 회사 생성 요청의 `companyMemo`는 `Company` 테이블 컬럼이 아니라 `CompanyMemoLog`의 첫 데이터로 저장한다.
- 회사 생성 요청의 `companyMemo`로 만들어지는 첫 메모 로그는 `memoType`을 `초기 메모`로 저장한다.
- `companyMemo`가 없으면 회사 생성 시 메모 로그를 만들지 않는다.
- 독립적인 회사 메모 로그 생성 API는 `memo`, `memoType`을 받는다.
- 회사별 사용자 비밀 메모는 `CompanyUserPrivateMemoLog` 테이블에 기록한다.
- 독립적인 회사 개인 비밀 메모 로그 생성 API는 `memo`만 받는다.
- 비밀 메모 원문은 데이터베이스에 평문으로 저장하지 않고 암호화해서 저장한다.
- 회사 개인 비밀 메모는 작성자 본인만 복호화된 내용을 볼 수 있고, 관리자도 원문을 볼 수 없다.
- 회사 메모 로그와 개인 비밀 메모 로그는 각각 10개씩 무한스크롤 방식으로 조회한다.
- 회사 메모 로그와 개인 비밀 메모 로그 수정 API는 `memo`만 수정한다.

회사 기본 도메인 1차 구현 당시에는 휴지통과 soft delete를 제외했다. 현재는 추가 유지보수로 회사 soft delete, 7일 이내 Trash 복구, 삭제 목록 조회가 구현되어 있다.

## 11. 제품

- UI 용어는 **제품**을 사용한다.
- 코드/데이터 모델 용어는 `Product`를 사용한다.
- `상품`, `영업상품` 표현은 정리 대상이다.
- 제품 단가는 선택 입력이다.
- 제품 분류는 기본 목록을 제공하되, 사용자가 직접 입력할 수 있다.

최신 Product 기본 도메인 1차 구현 범위는 `AGENT/PM_AGENT/DECISIONS/025_product_domain_basic_scope.md`를 우선한다.

1차 구현에서는 `Product`, `ProductCategory`, `ProductStatus`, `ProductMemoLog`, `ProductUserPrivateMemoLog`만 만들고, 제품가격은 `productPrice` 필수 정수로 받았다. 당시 `ProductConnection`, `ProductLog`, 휴지통, soft delete, `unitPrice`, `currency`, `description`은 범위에서 제외했다. 현재는 추가 유지보수로 제품 soft delete, 7일 이내 Trash 복구, 삭제 목록 조회가 구현되어 있다.

제품은 다음과 모두 연결 가능하다.

- 회사
- 담당자
- 딜

제품 연결에는 연결 타입을 둔다.

예:

- 관심 제품
- 기존 납품 제품
- 제안 중 제품
- 경쟁 제품
- 유지보수 대상 제품
- 기타

## 12. 딜

- 딜 금액은 필수 입력이다.
- 제품 단가가 있으면 딜 생성 시 참고값/기본값으로 불러올 수 있다.
- 딜 금액은 KRW 기본이다.
- 다른 통화는 추후 확장한다.

MVP 기본 딜 단계:

- 초기 접촉
- 협의중
- 성사
- 실패

딜 단계는 자유롭게 앞뒤 변경 가능하다. 단, 모든 변경은 딜 활동 로그에 자동 기록한다.

딜 가능성 표시는 다음 방식으로 한다.

- 기본: 긍정 / 중립 / 부정
- 고급 옵션: 숫자 퍼센트 입력

주간 일정 보고서 출력:

- 화면에서 주간표로 보기
- PDF 다운로드
- Excel 다운로드

## 16. 회의록

회의록은 MVP에 포함한다.

MVP 범위:

- 사용자가 텍스트로 회의 내용 입력
- AI 회의록 생성
- 사용자가 수정 후 저장
- 딜 연결
- 딜 활동 로그 자동 생성

MVP 제외:

- 음성 녹음
- STT

MVP 회의록 AI 결과 항목은 고정 9개다.

- 날짜
- 회사
- 담당자
- 부서
- 품목
- 진행단계
- 상세내용
- 향후계획
- 필요액션

추후 사용자가 항목/템플릿을 커스터마이즈 가능하게 확장한다.

회의록은 딜 없이도 저장 가능하며, 나중에 딜과 연결할 수 있다.

회의록 연결 방식:

- 작성 시 회사/담당자 직접 선택 가능
- AI가 회의 내용에서 회사/담당자를 추출해 후보 제안
- 딜 연결 시 딜의 회사/담당자 정보 자동 상속

## 17. 명함 OCR

명함 스캔/OCR은 MVP 필수다.

웹 MVP:

- 명함 이미지 업로드 기반 OCR
- User Web route는 `/business-cards`다. 기존 `/contacts/scan`은 legacy redirect로만 둔다.
- 사용자가 `명함스캔` 모달에서 이미지를 업로드하면 OpenAI OCR을 호출한다.
- 요청 중에는 `명함스캔 중` 진행 표시를 보여주고, 완료 전에는 사진 교체/삭제와 모달 닫기를 막는다.
- 성공 후에는 추출 결과를 회사명, 회사분야, 회사지역, 담당자명, 휴대폰, 이메일, 부서, 직급 입력값으로 보여주며 사용자가 확인/수정한다.
- 실패 시 같은 모달에서 재촬영/재시도 흐름을 제공하지 않고 실패 상태를 안내한다.
- 업로드 이미지는 저장하지 않는다.
- `BusinessCardScanLog`에는 성공/실패/확정 상태와 추출/보정값, provider model, prompt snapshot, token/cost metric, 처리 시간을 저장한다. 에러 상세는 이 테이블이 아니라 애플리케이션 로그에 남긴다.
- 명함 스캔 내역은 등록일 최신순 고정이며, 상태는 다중 필터와 `상태 초기화`를 제공한다.

모바일 앱 출시 후:

- 카메라 촬영 기반 명함 스캔

OCR 결과는 항상 사용자가 확인/수정 후 저장한다. 신뢰도가 높아도 자동 저장하지 않는다.

명함 OCR 저장 시 회사 처리:

- 회사명 기준으로 사용자 소유의 기존 회사를 재사용한다.
- 기존 회사가 없으면 새 회사를 생성한다.
- 담당자는 같은 회사 안에서 휴대폰 또는 이메일 기준으로 기존 담당자를 재사용한다.
- 기존 담당자가 없으면 새 담당자를 생성한다.
- 담당자는 반드시 회사에 소속되며, 회사 소속 없는 담당자 저장은 지원하지 않는다.
- 자동 매칭은 후보 제안과 확정 저장 보조까지만 한다.

## 18. Import / Export

Export는 범용 `ExportJob`로 처리하지 않고 각 도메인 목록 화면에서 동기 xlsx 다운로드로 처리한다.

현재 구현된 Export 대상:

- 회사: `GET /api/companies/export/xlsx`, User Web 표시 문구 `엑셀 다운로드`
- 담당자: `GET /api/contacts/export/xlsx`, User Web 표시 문구 `엑셀 다운로드`
- 제품: `GET /api/products/export/xlsx`, User Web 표시 문구 `엑셀 다운로드`
- 딜: `GET /api/deals/export/xlsx`, User Web 표시 문구 `엑셀 다운로드`

도메인 구분은 버튼 문구가 아니라 사용자가 보고 있는 목록 화면과 호출 API로 판단한다.

현재 목록 검색어와 필터는 export 요청에 반영하고, `page`는 export 요청에서 제외한다.

Export 제외 또는 후속:

- `/api/exports` 기반 범용 Export job
- `ExportJob` table
- 일정/회의록 export
- 주간 일정 보고서 PDF/Excel
- 민감 데이터 포함 선택 export

현재 구현된 Import 대상:

- 회사
- 담당자
- 제품
- 딜

현재 Import 방식:

- AI가 Excel/CSV 컬럼을 자동 매핑한다.
- 사용자가 매핑 결과를 확인/수정한다.
- 확정 후 Import한다.
- 딜 불러오기는 회사명, 담당자명, 제품명 매칭을 기준으로 딜과 연결 row를 함께 생성한다. 기존 항목이 없으면 사용자 보정값을 받아 회사/담당자/제품을 생성할 수 있다.

현재 구현 API:

- `GET /api/import-templates/active`
- `GET /api/import-templates/:templateId/download`
- `POST /api/imports`
- `GET /api/imports/:importJobId`
- `POST /api/imports/:importJobId/map`
- `PATCH /api/imports/:importJobId/mapping`
- `POST /api/imports/:importJobId/confirm`
- `GET /api/import-user-logs`
- `GET /api/import-user-logs/:importUserLogId`

Import 후속:

- 확정 전 ImportJob DB 영속화
- 서버 재시작 후 임시 job 이어받기

일정/회의록 Import는 MVP 제외다.

## 19. 민감 데이터

민감 데이터 기준:

- Memo 기록
- 회의록 본문
- 딜 금액
- 사용자가 `민감`으로 표시한 모든 데이터

Admin에서는 민감 데이터가 기본 마스킹된다.

원문 조회가 필요한 경우:

- 명시적 액션 필요
- 사유 입력 필요
- 감사 로그 기록 필수

민감정보 저장 경고는 MVP에 포함한다.

- 저장 전 경고 표시
- 사용자가 확인 후 저장
- 키워드/패턴 자동 감지는 MVP에서 하지 않음

도메인에 대한 사용자의 주관적 생각/판단 기록은 **Memo 기록**으로 부른다.

`오프더레코드` 표현은 MVP UI에서 사용하지 않는다.

## 20. 삭제/휴지통

- 삭제 시 즉시 완전 삭제하지 않는다.
- 7일 동안 일반 휴지통에서 복구할 수 있다.
- 7일이 지나면 일반 휴지통 목록에서 보이지 않는다. 유료 복구 정책과 시스템 정리 작업은 후속 범위다.
- MVP 1차에서 사용자/Admin 즉시 완전 삭제 API와 UI는 제공하지 않는다.
- 복구 기간 만료 전 알림은 후속 발송 또는 준비 항목으로 둔다.

대상:

- 회사
- 담당자
- 제품
- 딜
- 회의록
- 관련 Log/Memo 기록

일정은 현재 hard delete이며 휴지통 대상이 아니다.

## 24. 모노레포 구조

모노레포 루트는 저장소 루트다.

기존 `AI_INFRA` 폴더는 한 단계 감싸는 구조였으므로 제거하고, 내부 항목을 저장소 루트로 올린다.

루트에는 `package.json`을 두지 않는다.
workspace 설정도 사용하지 않는다.

FE와 BE는 모노레포 안에 있지만 패키지/의존성은 공유하지 않는다.

최종 구조:

```text
AGENT/
  PM_AGENT/
  UXUI_AGENT/
  SOFTWARE_AGENT/
    FRONT_AGENT/
    BACKEND_AGENT/
    DB_SCHEMA/
FE/
  user-web/
  admin-web/
BE/
archive/
```

AGENT 역할 구조:

- `PM_AGENT`: 제품 기획, MVP 범위, 도메인 개념, 문서 운영, PM 결정
- `UXUI_AGENT`: 사용자 흐름, 화면 구조, UX/UI 방향, UX/UI 결정
- `SOFTWARE_AGENT/FRONT_AGENT`: User Web/Admin Web 아키텍처, 코드 컨벤션, E2E, 배포, Frontend 결정
- `SOFTWARE_AGENT/BACKEND_AGENT`: Backend 아키텍처, API, 코드 컨벤션, 테스트, 배포, Backend 결정
- `SOFTWARE_AGENT/DB_SCHEMA`: DB schema와 테이블 설명

FE 하위:

- `FE/user-web`
- `FE/admin-web`

BE:

- 단일 NestJS 서버
- User API: `/api/*`
- Admin API: `/admin/api/*`

모바일 폴더는 지금 만들지 않는다. 웹 MVP 이후 모바일 개발 시 추가한다.

## 25. 테스트 자동화

현재 테스트 자동화의 우선순위는 User Web 핵심 업무 흐름이다. Admin Web은 관리자 페이지 본 구현 전까지 로그인/권한 확인 smoke만 유지한다.

원칙:

- User Web E2E는 `FE/user-web`에서 관리한다.
- Admin Web E2E는 `FE/admin-web`에서 관리하되, 현재는 `/admin/api/me` 기반 보호 라우트 검증 범위로 제한한다.
- 루트 공용 패키지나 workspace를 만들지 않는다.
- Playwright를 각 FE 앱의 E2E 도구로 사용한다.
- 백엔드는 위험도 높은 도메인/권한/민감정보 흐름을 중심으로 단위/통합 테스트를 둔다. Import는 업로드/매핑/검증/확정 흐름을 테스트하고, Admin 운영 조회는 후속 구현 시 테스트 범위를 추가한다.

우선 E2E 대상:

- 로그인/보호 라우트
- 회사/담당자/제품 CRUD
- 딜 생성/단계 변경/활동 로그
- 일정 CRUD와 연결
- 회의록 저장/딜 연결
- 회사/담당자/제품/딜 도메인별 xlsx 다운로드
- 휴지통 목록/상세/복구
- Admin login/role guard smoke

CI 실행 타이밍:

- PR마다 User Web 핵심 smoke E2E와 Admin auth smoke E2E를 실행한다.
- `main` merge 후 User Web 전체 E2E를 실행한다.
- Admin 운영 화면 전체 E2E는 관리자 페이지 본 구현 후 CI 범위에 추가한다.

- `production`

`staging` 환경은 MVP에서 제외한다.

## 27. UX/UI 첫 화면

로그인 후 첫 화면은 딜 파이프라인 중심으로 한다.

이유:

- 이 앱의 핵심은 개인 영업자가 현재 진행 중인 딜 상태를 빠르게 보는 것이다.
- 사용자는 딜 단계, 금액, 가능성, 다음 행동을 먼저 확인해야 한다.
- 일정, 알림, 빠른 입력, 요약 지표는 중요하지만 홈의 1순위는 아니다.

홈 화면 우선순위:

1. 진행 중인 딜 파이프라인
2. 딜 단계별 상태
3. 금액/가능성/다음 행동/마감 임박 표시
4. 오늘 일정과 후속 연락
5. 빠른 생성 버튼

딜 파이프라인 UI 방향:

- 참고 파일: `example.html`
- 참고 파일은 방향성만 사용하고 그대로 구현하지 않는다.
- 기본 desktop UI는 순수 칸반이 아니라 단계 탭 + 리스트/테이블 + 우측 상세 패널에 가깝게 한다.
- 모바일은 리스트/카드형으로 전환한다.
- 토스 UX/UI처럼 심플하고 깔끔한 정보 위계를 선호하되, 영업 도구에 필요한 정보 밀도는 유지한다.
- 가능성은 기본적으로 `긍정 / 중립 / 부정`으로 표시한다. 퍼센트 UI는 고급 옵션으로만 둔다.
- 민감 가능 Memo UI 용어는 `Memo 기록`을 사용한다.
- 베이지/크림 계열이 화면 전체를 지배하지 않게 한다.

UI 밀도:

- 중간 밀도로 간다.
- 토스식 단순함과 여백은 유지한다.
- 한 화면에서 여러 딜을 비교할 수 있는 정보량은 확보한다.
- 너무 sparse한 소비자 앱 화면도 피하고, 너무 촘촘한 ERP형 화면도 피한다.

메인 색상:

- 화이트/그레이 기반에 블루 중심으로 간다.
- 이유는 신뢰감을 주기 위해서다.
- 블루는 주요 액션, 선택된 내비게이션, 활성 탭, 포커스 상태에 사용한다.
- 상태 색상은 보조적으로만 사용한다.
  - 긍정/성사/완료: 그린
  - 주의/후속 필요: 앰버
  - 실패/지연/위험: 레드
- 네이비/다크톤이 화면을 지배하지 않게 한다.
- 베이지/크림톤이 화면을 지배하지 않게 한다.

글자 크기와 가독성:

- 중간으로 간다.
- 30~50대 사용자가 반복해서 봐도 피로하지 않게 한다.
- 너무 작은 ERP형 테이블 글자는 피한다.
- 너무 큰 소비자 앱형 글자도 피한다.
- 중요한 값은 명확히 보이게 하고, 보조 정보는 작게 두되 대비를 충분히 확보한다.

딜 리스트 표시 순서:

```text
딜이름 -> 회사/담당자 -> 단계 -> 금액 -> 가능성 -> 다음 행동 -> 마감일
```

모바일 카드 표시 순서:

```text
딜이름
회사/담당자
단계 · 금액 · 가능성
다음 행동
마감일
```

이유:

- 개인 영업자는 먼저 이 딜이 무엇이고 누구와 하는 건지 알아야 한다.
- 그 다음 현재 단계, 금액, 가능성을 비교한다.
- 마지막으로 다음 행동과 마감일을 확인한다.

모바일 딜 파이프라인:

- 단계 탭 + 카드형 리스트로 간다.
- 상단 탭은 `전체 / 초기 접촉 / 협의중 / 성사 / 실패`를 사용한다.
- 모바일 기본 UI로 테이블이나 가로 칸반을 사용하지 않는다.
- 카드에서는 다음 행동과 마감일이 바로 보여야 한다.

입력 UX:

- 빠른 등록은 모달로 한다.
- 상세 등록/수정은 별도 상세 페이지에서 한다.
- 회사/담당자/제품/딜은 처음부터 모든 필드를 강제하지 않는다.
- 최소 필드로 빠르게 저장하고, 상세 페이지에서 추가 정보를 보강한다.

빠른 등록 의도:

- 현장에서 입력을 미루지 않게 한다.
- 목록/파이프라인 맥락을 유지한다.
- 복잡한 입력은 좁은 모달에 억지로 넣지 않는다.

딜 빠른 등록 중 회사/담당자/제품이 없을 때:

- 딜 등록 모달 안에서 없는 항목을 최소 정보로 바로 생성한다.
- 검색 결과가 없으면 `새 회사 만들기`, `새 담당자 만들기`, `새 제품 만들기`를 보여준다.
- 생성 후 해당 항목을 딜에 자동 선택한다.
- 자유 텍스트로만 딜에 저장하는 방식은 사용하지 않는다.
- 중복 방지를 위해 기존 후보를 먼저 보여준다.

## 28. UX/UI 검색

검색 UX는 상단 통합검색과 화면별 검색/필터를 함께 사용한다.

상단 통합검색:

화면별 검색/필터:

- 현재 화면의 목록만 필터링한다.
- 딜 목록에서는 단계, 금액, 가능성, 다음 행동, 마감일, 회사/담당자로 필터링한다.

모바일:

- 통합검색은 전체 화면 검색 시트로 열 수 있다.
- 화면별 필터는 필터 시트로 열 수 있다.

## 30. UX/UI 일정/회의록 연결

일정과 회의록은 딜 중심으로 연결하되, 각 기능 화면에서도 생성/연결할 수 있게 한다.

원칙:

- 딜 상세에서 일정/회의록을 만들면 딜의 회사/담당자 정보를 기본 상속한다.
- 일정은 딜/회사/담당자와 연결할 수 있다.
- 회의록은 딜 없이 저장 가능하며, 나중에 딜과 연결할 수 있다.
- 회의록이 딜과 연결되면 딜 활동 로그가 자동 생성된다.
- AI 회의록은 회사/담당자/딜 후보를 제안하고, 사용자가 확인 후 저장한다.

## 32. UX/UI Admin 톤

원칙:

- 테이블 중심
- 필터 중심
- 운영 콘솔 느낌
- 민감정보 기본 마스킹
- 위험 액션은 사유 입력과 감사 로그 필수

Admin은 사용자 앱처럼 부드럽고 감성적인 화면보다 정확성, 추적성, 안전성을 우선한다.

## 33. 기획 검토 게이트

`AGENT`와 `TODO`에 작성된 기획서, 요구사항 문서, API 명세, DB 스키마, FE/BE 작업 문서는 구현 전에 기획 검토 게이트를 통과해야 한다.

검토 기준 문서:

- `AGENT/PM_AGENT/CONVENTION/PLANNING_REVIEW_CHECKLIST.md`

적용 원칙:

- 새 `TODO/{PLAN_NAME}` 계획 폴더를 만든 뒤 실제 구현 전에 검토한다.
- 사용자가 "검토해줘", "구현 들어가도 되는지", "부족한 점이 있는지"라고 요청하면 이 기준으로 검토한다.
- 검토 결과는 `통과`, `조건부 통과`, `수정 필요`, `보류` 중 하나로 판정한다.
- Critical 문제가 있으면 구현을 시작하지 않는다.
- 사용자의 결정이 필요한 항목은 AI가 임의로 확정하지 않는다.

## 34. TODO COMMON 공통 계약 구조

앞으로 `TODO` 아래에 특정 요구사항 또는 기획 계획 폴더를 만들 때는 `COMMON` 폴더를 반드시 둔다.

`COMMON`은 Frontend와 Backend가 함께 봐야 하는 공통 계약 문서를 관리한다.

기본 구조:

```text
TODO/
  {PLAN_NAME}/
    README.md
    COMMON/
      README.md
      USER-FLOW.md
      GOAL-WORK-ORDER.md
      PLANNING-REVIEW.md
      API-SPEC/
      GOAL-SPECS/
    FE-TODO/
    BE-TODO/
      DB-SCHEMA.md
```

적용 원칙:

- 사용자 흐름은 `COMMON/USER-FLOW.md`에 둔다.
- `/goal` 우선순위 작업 순서는 `COMMON/GOAL-WORK-ORDER.md`에 둔다.
- API 명세는 `COMMON/API-SPEC`에 둔다.
- API 명세는 전체를 한 번에 쓰지 않고 `/goal` 우선순위에 맞춰 나눈다.
- 각 `/goal`별 화면 명세, API 연결, DB 연결, 완료 기준은 `COMMON/GOAL-SPECS`에 둔다.
- 구현 전 기획 검토 결과는 `COMMON/PLANNING-REVIEW.md`에 둔다.
- DB 스키마는 `BE-TODO/DB-SCHEMA.md`에 구현 직전 수준으로 작성하되, API 명세와 goal 명세에서 연결한다.
- 이 구조는 사용자가 확정한 기본값이므로 같은 기획 문서를 만들 때 다시 질문하지 않는다.

관련 결정:

- `AGENT/PM_AGENT/DECISIONS/018_todo_common_contract_structure.md`

## 35. Goal 완료 후 검토와 TODO_LOG 기록

특정 `/goal` 또는 명시적인 작업 단위가 끝나면 AI 작업자는 다음 작업으로 넘어가기 전에 사용자에게 검토 여부를 질문한다.

질문 문구:

```text
이번 작업에 대해 검토를 해드릴까요?
```

검토가 `통과` 또는 사용자가 수용한 `조건부 통과` 상태가 되면 `TODO_LOG/YYYY-MM-DD/` 아래에 작업 기록 문서를 작성한다.

TODO_LOG 기록에는 다음을 포함한다.

- 작업명
- 작업 일자
- 관련 계획과 goal
- 적용 범위
- 변경 파일
- 검증 결과
- 검토 결과
- 남은 리스크 또는 보류 사항
- 다음 권장 작업

검토 없이 TODO_LOG를 먼저 작성하지 않는다. 실패한 작업을 통과한 작업처럼 기록하지 않는다. 비밀값, token, 개인정보 원문은 TODO_LOG에 기록하지 않는다.

관련 결정:

- `AGENT/PM_AGENT/DECISIONS/022_goal_completion_review_todo_log.md`

## 36. 시간/타임존 기준

DB/API/Frontend의 시간 처리 기준은 `AGENT/PM_AGENT/DECISIONS/026_time_timezone_policy.md`와 `AGENT/SOFTWARE_AGENT/DB_SCHEMA/TIME_AND_TIMEZONE_POLICY.md`를 따른다.

기본값:

- `createdAt`, `updatedAt` 같은 시스템 시각은 UTC 기준으로 저장한다.
- 일정의 `startAt`, `endAt`은 사용자가 입력한 local date-time과 IANA `timeZone`을 해석한 뒤 DB에는 UTC instant로 저장한다.
- 사용자가 입력한 현지 날짜/시간을 저장하는 업무 테이블은 `timeZone` 컬럼을 함께 둔다.
- 사용자 화면 표시는 일정/사용자/조직 timezone 기준으로 표시한다.
- 날짜만 필요한 값은 Prisma `DateTime @db.Date`를 사용한다.
