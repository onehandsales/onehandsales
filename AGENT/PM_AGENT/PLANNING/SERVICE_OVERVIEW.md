# 서비스 기획 개요

> 문서 목적: 이 문서는 `한손에 영업 / onehand.sales`가 기획상 어떤 서비스인지 한 번에 이해하기 위한 입문 문서다.

## 1. 한 문장 정의

`한손에 영업 / onehand.sales`는 개인 영업자가 회사, 담당자, 제품, 딜, 일정, 회의록을 한곳에서 연결해 관리하는 B2C 개인 영업 업무 도구다.

회사나 팀이 도입하는 무거운 CRM이 아니라, 영업자 개인이 직접 쓰는 경량 업무 도구로 시작한다.

## 2. 누구를 위한 서비스인가

주 사용자는 30~50대 개인 영업자다.

특히 다음과 같은 사용자를 기본 대상으로 본다.

- 회사 시스템이 있어도 개인 메모와 실제 영업 맥락을 따로 관리하는 영업자
- 담당자, 회사, 제품, 딜, 일정을 엑셀, 메모장, 캘린더, 명함에 흩어져 관리하는 영업자
- 오늘 어떤 딜을 챙겨야 하는지, 다음 행동이 무엇인지 빠르게 확인해야 하는 영업자
- 주간 일정 보고서나 회의 후 정리를 매번 다시 만드는 영업자
- 대형 CRM보다 빠른 입력, 빠른 검색, 개인 기준 정리를 더 중요하게 느끼는 영업자

초기 가격 가설은 월 5,900~6,900원 구독이다. MVP에는 결제를 붙이지 않고, MVP 이후 초기 유료 운영은 계좌이체 수동 처리로 시작한다.

초기 판매/검토 국가는 한국, 일본, 대만, 미국, 영국, 싱가포르, 호주, 캐나다로 본다. 공개/인증 화면은 `ko`, `ja`, `zh-tw`, `en-us`, `en-gb`, `en-sg`, `en-au`, `en-ca` URL locale을 지원하고, 로그인 이후 `/app` 관리 화면은 한국어 우선으로 운영한 뒤 나중에 전체 다국어화를 진행한다.

## 3. 해결하려는 문제

개인 영업자는 실제 업무에서 많은 정보를 여러 곳에 나누어 저장한다.

- 회사 정보는 회사 시스템이나 엑셀에 있다.
- 담당자 정보는 명함, 휴대폰 연락처, 메모에 흩어져 있다.
- 제품과 제안 내용은 별도 파일이나 기억에 의존한다.
- 딜 금액, 단계, 가능성, 다음 행동은 개인 메모로 관리되는 경우가 많다.
- 일정은 캘린더에 있지만 딜, 회사, 담당자와 충분히 연결되지 않는다.
- 회의 후 기록은 남겨도 딜 활동 로그와 이어지지 않는다.

이 서비스는 이 흩어진 정보를 개인 영업자의 실제 업무 기준으로 연결한다.

## 4. 핵심 가치

이 서비스가 제공하는 핵심 가치는 네 가지다.

첫째, 영업 정보를 한곳에 모은다.

회사, 담당자, 제품, 딜을 각각 따로 보관하지 않고 서로 연결해 관리한다. 사용자는 특정 회사나 담당자를 볼 때 관련 딜, 제품, 메모, 회의록까지 함께 확인할 수 있다.

둘째, 현재 진행 중인 딜을 빠르게 파악한다.

딜이름, 회사, 담당자, 단계, 금액, 다음 행동, 예상 마감일을 중심으로 보여준다. 사용자는 지금 어떤 딜이 막혀 있고 무엇을 해야 하는지 빠르게 판단할 수 있다.

셋째, 일정과 회의록을 영업 흐름에 연결한다.

일정은 딜과 연결되고, 회의록은 회사/담당자/제품/딜 snapshot과 함께 저장된다. 회의록 AI/STT 초안은 자동 저장이 아니라 사용자가 확인하고 수정한 뒤 저장하는 흐름을 따른다.

넷째, 입력 부담을 줄인다.

빠른 등록, 명함스캔 OCR 자동 입력, 화면별 검색/필터, 통합검색, 회의록 AI/STT 정리, 데이터 불러오기 AI 매핑, 도메인별 엑셀 다운로드 같은 기능으로 처음 입력하거나 다시 찾는 부담을 줄이는 방향이다. 명함스캔은 자동 입력 후 사용자가 확인/수정하고 저장하는 흐름이며, 데이터 불러오기는 회사/담당자/제품/딜 CSV/XLSX 업로드 후 AI 매핑과 사용자 보정을 거쳐 확정 저장하는 흐름이다.

## 5. 사용자가 경험하는 핵심 루프

서비스의 MVP 핵심 루프는 다음 순서다.

1. 사용자가 회사, 담당자, 제품을 등록한다.
2. 등록된 회사, 담당자, 제품을 기반으로 딜을 만든다.
3. 딜 단계, 금액, 다음 행동, 메모를 관리한다.
4. 일정을 딜과 연결하고 일정 화면에서 확인한다.
5. 회의 내용을 입력하거나 음성/텍스트 기반 AI 초안으로 회의록을 만든다.
6. 회의록을 확인, 수정, 저장하고 필요한 딜과 연결한다.
7. 이후 통합검색과 상세 화면에서 관련 정보를 빠르게 다시 찾는다.

이 루프의 목적은 영업자가 매일 해야 할 다음 행동을 잊지 않고, 딜 맥락을 잃지 않게 하는 것이다.

## 6. 주요 도메인

서비스의 기본 도메인은 여섯 개다.

### 회사

영업 대상 회사다. 회사명, 분야, 지역, 일반 메모, 개인 비밀 메모를 관리한다. 회사 상세에서는 연결된 담당자와 딜을 확인할 수 있다.

### 담당자

회사에 소속된 사람이다. 담당자는 회사 없이 독립적으로 존재하지 않는다. 이름, 휴대폰, 이메일, 부서, 직급, 메모를 관리하고 연결된 딜을 확인한다.

### 제품

영업자가 제안하거나 판매하는 제품이다. 제품명, 가격, 카테고리, 상태, 메모를 관리한다. 제품은 딜과 다대다로 연결될 수 있다.

### 딜

실제 영업 건이다. 회사, 담당자, 제품, 금액, 단계, 다음 행동, 예상 종료일을 가진다. 이 서비스의 가장 중요한 업무 단위다.

### 일정

영업 일정이다. 일정은 여러 딜과 연결될 수 있으며, 사용자 timezone 기준으로 월간/주간 화면에서 확인한다.

### 회의록

회의 내용을 저장하는 기록이다. 회사, 담당자, 제품, 딜과 연결 snapshot을 남긴다. AI 없이 직접 작성 후 바로 저장할 수 있고, 필요할 때 텍스트 AI 초안 또는 STT+AI 초안으로 내용을 정리할 수 있다. 어떤 경로든 최종 저장은 사용자가 확인한 뒤 수행한다.

## 7. 화면 방향

공개/인증 canonical URL은 locale prefix를 사용한다. 예: `/ko`, `/ko/login`, `/ja/signup`, `/en-us/pricing`. 기존 `/`, `/login`, `/signup`, `/pricing`, `/contact`, `/about`, `/security`, `/terms`, `/privacy`는 선호 locale URL로 redirect한다. 로그인 후 `/app`은 오늘의 업무를 보여주는 홈 대시보드이며, `/app/*`에는 locale prefix를 붙이지 않는다.

홈에서는 오늘 일정, 진행 딜, 마감 임박 딜, 최근 회의록, 빠른 실행을 보여준다. 고밀도 딜 비교와 수정은 `/app/deals`에서 처리한다.

`/app/deals`는 핵심 영업 작업 화면이다.

데스크톱에서는 단계 탭, 리스트/테이블, 우측 상세 패널 구조를 사용한다. 모바일에서는 단계 탭과 카드형 리스트를 사용한다. 모바일 기본 UI로 테이블이나 가로 칸반을 쓰지 않는다.

상단 통합검색은 회사, 담당자, 제품, 딜, 일정, 회의록을 한 번에 찾는 기능이다. 각 화면의 검색/필터는 현재 화면 목록만 좁혀보는 역할을 한다.

Admin Web은 후속 단계에서 만들 데스크톱 전용 운영 콘솔이다. 구현 시 데이터 테이블, 필터, 서버 페이지네이션, 민감정보 마스킹, 감사 로그를 중심으로 한다.

## 8. MVP에 포함되는 것

현재 MVP 방향에는 다음이 포함된다.

- 소셜 인증: 현재 구글만 활성화한다. 애플은 iOS 대응 시, 라인은 일본/대만 확장 시 별도 구현한다.
- 공개/인증 화면 URL locale: `ko`, `ja`, `zh-tw`, `en-us`, `en-gb`, `en-sg`, `en-au`, `en-ca`
- 회사, 담당자, 제품 관리
- 딜 관리와 다음 행동/메모 로그
- 일정 관리와 딜 연결
- 일정 목록/캘린더 화면
- 수동 회의록 관리
- 회의록 텍스트 AI 초안
- 회의록 STT+AI 초안
- 회의록 저장 후 딜 연동과 딜 활동 로그 생성
- 상단 통합검색
- 명함스캔 OCR 자동 입력, 진행 표시, 확인/수정 후 회사/담당자 저장
- 회사/담당자/제품/딜 데이터 불러오기
- 회사/담당자/제품/딜 도메인별 Excel export
- 민감정보 기본 보호 정책
- 휴지통 목록/상세/7일 이내 복구

## 10. 현재 구현 관점의 요약

2026-07-10 기준으로 Backend는 Auth/User, Company, Contact, BusinessCard OCR, Product, Deal, Schedule, MeetingNote 수동 도메인, Search, Trash, DataImport, MeetingNote AI/STT draft API, 회의록 저장 후 딜 연동 API를 구현한 상태다.

User Web은 URL locale 기반 공개/인증 진입면과 `/app` 홈 대시보드, 회사, 담당자, 명함 스캔, 제품, 딜, 일정, 수동 회의록, 회의록 AI/STT 초안, 회의록 저장 후 딜 연동 화면, 회사/담당자/제품/딜 데이터 불러오기, 회사/담당자/제품/딜 엑셀 다운로드, 상단/모바일 통합검색, 휴지통 목록/상세/복구가 실제 API와 연결되어 있다. 명함 스캔 화면은 등록일 최신순 내역, 상태 다중 필터, 이미지 업로드 기반 명함스캔 모달을 제공한다. Import preview validation은 누락된 셀에만 메시지를 표시한다.

Admin Backend는 현재 `GET /admin/api/me`만 구현되어 있으므로, Admin 페이지와 운영 조회 화면은 후속 구현 범위로 본다.

2026-07-10 기준 핵심 업무 happy path, URL locale smoke, API/security smoke, BE/FE/admin-web 자동 점검은 통과했다. 출시 전 남은 품질 범위는 UX/UI 공통 QA, 모바일 브라우저 QA, Chrome/Edge QA, 다중 계정 보안 QA, DB/운영 환경 정합성 확인이다.

## 11. 이 서비스를 이해할 때 중요한 기준

이 서비스는 CRM을 그대로 작게 만든 제품이 아니다.

핵심은 개인 영업자의 현장 업무 흐름이다. 사용자가 빨리 입력하고, 오늘 할 일을 확인하고, 딜 맥락을 잃지 않고, 회의와 일정을 딜로 다시 연결하는 것이 중요하다.

따라서 화면은 마케팅 사이트처럼 설명적이거나 장식적이면 안 된다. 업무 도구처럼 조용하고 읽기 쉬워야 하며, 딜과 다음 행동을 빠르게 비교할 수 있어야 한다.

또한 이 서비스의 데이터는 사용자 개인 기준으로 소유된다. User API는 항상 본인 데이터만 다루고, Admin API는 별도 경로와 권한, 민감정보 마스킹, 감사 로그 기준을 따라야 한다.
