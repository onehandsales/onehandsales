# Next Feature Priorities

기준일: 2026-07-07

이 문서는 `onehand.sales`의 다음 작업 우선순위를 정리한다. 현재 기준에서는 새 기능 개발보다 `MVP Baseline QA`, 진입/인증/라우팅 안정화, 반응형/모바일 브라우저 확인이 먼저다. UX/UI 기준은 Notion식 작업도구 UX를 1차 reference로 삼는다.

## 1. 현재 제품 범위

현재 제품 범위는 다음과 같다.

- Web
- 반응형 Web
- 모바일 브라우저 Web

현재 범위가 아닌 항목은 다음과 같다.

- iOS/Android 네이티브 앱
- 네이티브 푸시
- 네이티브 연락처/캘린더 연동
- 오프라인 앱 동작
- 앱스토어/플레이스토어 배포

따라서 모바일 QA는 네이티브 앱 QA가 아니라, 현재 Web 제품이 모바일 브라우저에서도 핵심 업무를 수행할 수 있는지 확인하는 QA다.

## 2. 지금 부족한 부분

### 2.1 MVP Baseline QA 미완료

핵심 기능은 구현되어 있지만, 실제 사용자가 처음부터 끝까지 업무를 완료할 수 있는지 검증 기록이 부족하다.

확인해야 할 기준 흐름:

1. 로그인
2. 회사 생성
3. 담당자 생성
4. 제품 생성
5. 딜 생성
6. 일정 생성
7. 회의록 생성
8. 검색
9. 목록/필터/정렬
10. 수정
11. 도메인별 XLSX 다운로드
12. 삭제
13. 휴지통 복구
14. 복구 데이터 재확인

### 2.2 진입/인증/라우팅 정합성 부족

랜딩과 푸터를 고도화하기 전에 사용자가 제품으로 진입하는 최소 경로가 안정적이어야 한다.

부족한 항목:

- `/`, `/login`, `/app` 역할 정리 필요
- 로그인 전 보호 라우트 차단 확인 필요
- 로그인 후 `/app` 진입 확인 필요
- 새로고침 후 세션 유지 정책 확인 필요
- 로그아웃 후 보호 화면 차단 확인 필요
- 실제 기준 라우트인 `/app/*`와 문서/E2E 기대값 정합성 확인 필요
- 모바일 하단 홈 링크가 `/`가 아니라 `/app` 기준으로 동작하는지 확인 필요

### 2.3 자동 검증 기준선 일부 미확정

BE 단위 테스트와 타입체크는 통과했지만, 전체 릴리즈 판단에 필요한 검증 기록은 아직 부족하다.

필요한 기록:

- BE `typecheck`, `lint`, `test`, `build`
- FE/user-web `typecheck`, `lint`, `build`
- FE/user-web E2E가 현재 구현 범위와 맞는지 확인
- FE/admin-web은 현재 QA 범위 제외 또는 `typecheck/build` 정도만 선택 확인
- PowerShell에서 `pnpm` 실행 정책 문제가 있으면 `pnpm.cmd` 기준으로 기록

### 2.4 반응형/모바일 브라우저 검증 부족

현재는 네이티브 앱이 없으므로 모바일 브라우저 Web도 제품 범위다. “모바일 앱에서 나중에 해결”로 넘기면 안 된다.

확인해야 할 항목:

- 1440px 데스크톱
- 1280px 노트북
- 768px 태블릿
- 390px 모바일
- 360px 모바일
- 로그인/홈/딜/일정/회의록/입력 폼/모달/하단 내비게이션
- 키보드가 올라온 상태에서 입력 필드와 저장 버튼 사용 가능 여부
- 긴 텍스트, 긴 이메일, 긴 회사명/제품명/딜이름 레이아웃 깨짐 여부

### 2.5 외부 진입면 미완성

랜딩 페이지, 푸터, 약관/개인정보 같은 외부 진입면은 아직 부족하다.

다만 우선순위는 핵심 기능 QA 이후다. 지금 필요한 수준은 제품 진입을 막지 않는 최소 정리다.

최소 정리 기준:

- `/`의 역할 확정
- 푸터 노출 위치와 최소 링크 정리
- 약관/개인정보 placeholder 또는 정책 페이지 경로 정리
- 랜딩 화면이 실제 제품 진입을 방해하지 않음

### 2.6 환경 문서 부족

환경 파일 존재와 필수 변수를 확인할 수 있는 문서가 부족하다.

부족한 항목:

- `BE/.env.example` 없음
- `FE/user-web/.env.example` 없음
- 실제 `.env` 값은 문서에 기록하면 안 됨
- OpenAI/Supabase/JWT/private memo encryption 관련 필수 변수 목록 정리 필요

### 2.7 알려진 한계 정리 필요

현재 실패로 처리하지 않을 항목은 QA 결과에 명확히 `N/A` 또는 `Known limitation`으로 기록해야 한다.

알려진 한계:

- 확정 전 ImportJob은 서버 메모리에 저장되어 재시작 시 사라질 수 있음
- `/app/notifications`는 `/app`으로 리다이렉트됨
- `/app/export`는 `/app`으로 리다이렉트됨
- `/app/schedules/week`는 `/app/schedules`로 리다이렉트됨
- Admin 운영 화면은 현재 QA 범위에서 제외
- 알림, 결제, 구독은 현재 QA 범위에서 제외
- iOS/Android 네이티브 앱은 현재 제품 범위가 아님

### 2.8 Notion식 UX/UI 기준 미반영

앞으로 모든 UX/UI는 Notion식 작업도구 UX를 1차 기준으로 본다. 현재 화면은 기능 구현 중심으로 만들어져 있어, 전체 화면이 아직 일관된 `workspace + sidebar + page + database view + record detail` 구조로 정리되었다고 보기 어렵다.

부족한 항목:

- 전체 IA가 좌측 sidebar와 page 중심 구조로 정돈되어 있는지 확인 필요
- 회사/담당자/제품/딜/일정/회의록 목록이 database view처럼 읽히는지 확인 필요
- 상세 화면이 Notion page처럼 속성 영역과 본문 section으로 분리되어 있는지 확인 필요
- row/card 클릭이 record detail로 자연스럽게 이어지는지 확인 필요
- 필터, 정렬, 검색, view 상태가 화면마다 일관되는지 확인 필요
- 모바일 브라우저에서 table이 card/list로 전환되어도 같은 정보 구조가 유지되는지 확인 필요
- Notion 브랜드, 로고, 고유 문구, 고유 화면을 복제하지 않는 선을 지키는지 확인 필요

| 9 | 외부 진입면 최소 정리 | 제품 진입 신뢰도 확보 | 랜딩/푸터/정책 링크가 최소 수준으로 정리 |

## 4. QA 이후 기능 우선순위

아래 항목은 MVP Baseline QA와 S0/S1/S2 정리가 끝난 뒤 검토한다.

| 순서 | 작업 | 이유 |
| --- | --- | --- |
| 1 | DataImport Job 영속화 | 서버 재시작/배포 시 업로드, 매핑, 검증 중인 작업 유실 방지 |
| 2 | 검색/필터 고도화 | 데이터가 쌓일수록 탐색 효율이 중요해짐 |
| 3 | 주간 일정/영업 리포트 | 일정, 딜, 회의록 데이터를 영업 판단으로 연결 |
| 4 | 휴지통/삭제 정책 고도화 | 복구 기한, 만료, 장기 보관 정책 명확화 |
| 5 | 딜 가능성/확률 | 파이프라인 우선순위 판단 강화 |
| 6 | 범용 활동 타임라인 | 딜 중심 활동을 한 화면에서 추적 |
| 7 | 회의록 AI/STT 이력 고도화 | AI/STT 품질 분석, 실패 추적, 개인정보 정책 정리 |
| 8 | 반복 일정 / 외부 캘린더 가져오기 | 일정 도메인 확장 |
| 9 | 대용량/비동기 Export | 동기 XLSX 다운로드 한계 대응 |

## 5. 지금 바로 할 일

바로 다음 행동은 새 기능 개발이 아니다.

1. `QA_CHECKLIST.md` 기준으로 MVP Baseline QA를 시작한다.
2. QA 과정에서 Notion식 UX/UI 기준에 맞지 않는 화면 구조를 FAIL 또는 UX 개선 항목으로 기록한다.
3. QA 결과를 PASS/FAIL/BLOCKED/N/A로 기록한다.
4. 발견 버그를 S0/S1/S2/S3/S4로 분류한다.
5. S0/S1/S2만 먼저 수정한다.
6. 그 다음 Notion식 핵심 화면 개선, 랜딩/푸터, 환경 문서를 최소 정리한다.

이 순서가 끝나기 전에는 DataImport Job 영속화나 새 영업 기능을 시작하지 않는다.
