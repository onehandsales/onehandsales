# Next Feature Priorities

기준일: 2026-07-06

이 문서는 `한손에 영업 / onehand.sales`의 다음 기능 개발 우선순위를 정리한다.

관리자 페이지, 알림 기능, 과금 및 구독 결제는 현재 우선순위에서 제외한다.

## 1. 전제

- 현재 User Web 핵심 업무 흐름은 대부분 실제 API와 연결되어 있다.
- 남은 작업은 신규 기능보다 안정성, 데이터 내구성, 검색성, 영업 판단 고도화에 가깝다.
- 기능 개발 전에는 작은 라우팅/문서/환경 구성 정합성을 먼저 정리한다.
- DB 변경이 필요한 기능은 먼저 `TODO/<PLAN_NAME>/COMMON/API-SPEC`와 DB schema 계획을 작성한 뒤 구현한다.

## 2. 추천 우선순위

| 순서 | 작업 | 목적 | 우선 이유 |
| --- | --- | --- | --- |
| 1 | 작은 정합성 정리 | 모바일 하단 홈 링크, `/app/*` 라우팅 기준, env example 정리 | 개발 전 혼동을 줄이는 저비용 선행 작업 |
| 2 | DataImport Job 영속화 | 확정 전 import job을 DB에 저장 | 서버 재시작/배포 시 업로드, 매핑, 검증 중인 작업 유실 방지 |
| 3 | 검색/필터 고도화 | 회사/담당자/상품/딜을 더 정교하게 탐색 | 데이터가 쌓이면 사용 빈도가 가장 빠르게 증가 |
| 4 | 주간 일정/영업 리포트 | 주간 일정, 딜 활동, 회의록 연결 요약 | 기존 Schedule/Deal/MeetingNote 데이터 활용 가능 |
| 5 | 휴지통/삭제 정책 고도화 | 7일 이후 처리, 완전 삭제, 장기 보관 기준 정리 | 데이터 신뢰성과 운영 정책 안정화 |
| 6 | 딜 가능성/확률 | 딜의 긍정/중립/부정 또는 성공 확률 관리 | 파이프라인 우선순위 판단에 직접 도움 |
| 7 | 범용 활동 타임라인 | 딜 중심 활동을 한 화면에 통합 | 영업 현황 판단 화면의 밀도와 유용성 강화 |
| 8 | 회의록 AI/STT 이력 고도화 | STT 원문, AI 생성/재생성, provider 처리 이력 관리 | 품질 개선, 장애 분석, 개인정보 정책 정리 |
| 9 | 반복 일정 / 구글 캘린더 가져오기 | 반복 일정과 외부 캘린더 이벤트 수집 | 일정 도메인 확장 및 외부 연동 기반 |
| 10 | 대용량/비동기 Export | ExportJob 기반 비동기 다운로드 | 현재 동기 xlsx가 한계에 닿을 때 확장 |

최종 실행 순서:

```text
정합성 정리
→ DataImport Job 영속화
→ 검색/필터 고도화
→ 주간 일정/영업 리포트
→ 휴지통/삭제 정책 고도화
→ 딜 가능성/확률
→ 범용 활동 타임라인
→ 회의록 AI/STT 이력 고도화
→ 반복 일정 / 구글 캘린더 가져오기
→ 대용량/비동기 Export
```

## 3. DB 설계 필요 여부

DB 영향은 세 단계로 나눈다.

- `새 테이블 필요`: 구현 전에 테이블, 관계, 인덱스, 보존 정책을 설계해야 한다.
- `기존 테이블 변경`: 새 테이블보다는 기존 모델 컬럼, enum, 인덱스 변경이 중심이다.
- `DB 변경 불필요 또는 선택`: 우선 구현은 기존 데이터 조회/화면/문서 정리로 가능하다.

| 작업 | DB 영향 | 판단 |
| --- | --- | --- |
| 작은 정합성 정리 | DB 변경 불필요 | 라우팅, 문서, env example 정리 작업 |
| DataImport Job 영속화 | 새 테이블 필요 | 확정 전 job, row, mapping, validation 상태를 서버 메모리 대신 저장해야 함 |
| 검색/필터 고도화 | 기존 테이블 변경 또는 선택 | 우선은 기존 테이블 query/index 보강으로 가능. saved search, 검색 전용 read model이 필요해질 때만 새 테이블 검토 |
| 주간 일정/영업 리포트 | DB 변경 불필요 또는 선택 | 즉시 생성 리포트는 기존 Schedule/Deal/MeetingNote 조회로 가능. 저장된 리포트 이력, 공유, 다운로드 이력을 남기려면 새 테이블 필요 |
| 휴지통/삭제 정책 고도화 | 기존 테이블 변경 또는 선택 | 현재 soft delete 필드로 기본 정책 가능. 삭제 이력, 복구 요청, 장기 보관, legal hold를 관리하려면 새 테이블 필요 |
| 딜 가능성/확률 | 기존 테이블 변경 | `Deal`에 가능성 상태/확률 컬럼 추가가 중심. 변경 이력까지 저장하려면 별도 history table 검토 |
| 범용 활동 타임라인 | 새 테이블 필요 | 도메인별 로그를 안정적으로 통합 조회하려면 timeline/activity read model 또는 event table 설계 필요 |
| 회의록 AI/STT 이력 고도화 | 새 테이블 필요 | AI/STT 시도, provider, 원문/결과, 실패 사유, 재생성 이력을 별도 lifecycle로 관리해야 함 |
| 반복 일정 / 구글 캘린더 가져오기 | 새 테이블 필요 | recurrence rule, exception, 외부 캘린더 연결, 외부 event mapping 설계 필요 |
| 대용량/비동기 Export | 새 테이블 필요 | ExportJob 상태, 파일 위치, 만료, 요청 조건 snapshot 저장 필요 |

## 4. 진짜로 테이블 설계가 필요한 항목

다음 항목은 기능 구현 전에 DB 테이블 설계를 먼저 해야 한다.

1. DataImport Job 영속화
2. 범용 활동 타임라인
3. 회의록 AI/STT 이력 고도화
4. 반복 일정 / 구글 캘린더 가져오기
5. 대용량/비동기 Export

다음 항목은 새 테이블보다 기존 테이블 변경 또는 인덱스 설계가 우선이다.

1. 검색/필터 고도화
2. 휴지통/삭제 정책 고도화
3. 딜 가능성/확률

다음 항목은 초기 구현 기준으로 DB 테이블 설계가 필수는 아니다.

1. 작은 정합성 정리
2. 주간 일정/영업 리포트

단, 주간 리포트를 저장 가능한 보고서로 만들거나, PDF/Excel 다운로드 이력을 사용자에게 보여주려면 별도 report/export 이력 테이블이 필요하다.

## 5. 다음 실행 단위 제안

바로 실행 가능한 첫 작업:

1. 모바일 하단 홈 링크가 `/app` 기준으로 동작하는지 확인하고 수정한다.
2. `/app/*` 기준으로 문서와 E2E 라우팅 기대값을 정리한다.
3. `BE/.env.example`, `FE/user-web/.env.example`을 추가한다.
4. `DataImport Job 영속화` 계획 폴더를 만들고 DB/API/FE 영향 범위를 계약 문서로 먼저 작성한다.

`DataImport Job 영속화`부터는 DB migration이 필요한 작업이므로, Prisma schema와 API 계약을 먼저 확정한 뒤 구현한다.
