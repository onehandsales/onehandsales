# 기술 스택 및 아키텍처

> 2인 풀스택 + AI 활용 + 4주 출시 시나리오 기준
> **User용 웹 + iOS + Android + Admin 웹 = 4 클라이언트, 단일 백엔드**

## 1. 최종 기술 스택 결정

### 1.1 클라이언트 구성 (4 클라이언트)

| 클라이언트 | 대상 사용자 | 기술 | 호스트 |
|-----------|-----------|------|--------|
| **User 웹** | 영업사원 (노트북) | React + Vite | app.yourdomain.com |
| **iOS 앱** | 영업사원 (iPhone) | React Native (Expo) | App Store |
| **Android 앱** | 영업사원 (Android) | React Native (Expo) | Play Store |
| **Admin 웹** | 관리자 (PC) | React + Vite + TanStack Table + Recharts | admin.yourdomain.com |

**백엔드는 단일** (api.yourdomain.com):
- User 라우트: `/api/*`
- Admin 라우트: `/admin/api/*` (AdminGuard로 보호)

**왜 PWA가 아닌 네이티브 앱인가**:
- App Store / Google Play 스토어 배포 필요 (정식 비즈니스 신뢰도)
- iOS 푸시 알림 (PWA 푸시는 iOS 제한 큼)
- 카메라 UX (명함 스캔이 핵심) → 네이티브 카메라가 우수
- 오프라인 저장 (SQLite) 안정성
- 사용자가 "진짜 앱"으로 인식

**왜 React Native (Expo)인가**:
- 팀이 React 사용 → 지식 재사용 (Flutter는 Dart 새로 배움)
- Expo EAS Build로 빌드/배포 1일 셋업
- iOS + Android 단일 코드베이스
- TypeScript, TanStack Query, React Hook Form 그대로 재사용
- 카메라(`expo-camera`), SQLite(`expo-sqlite`), 푸시(`expo-notifications`) 검증된 라이브러리

**왜 Capacitor가 아닌가**:
- 웹뷰 기반이라 카메라/오프라인 UX가 네이티브만 못함
- 명함 스캔이 핵심인데 카메라 품질이 결정적

```
└── sales-shared-types/         # API 타입 공유 npm package (선택)
```

**왜 4 레포인가**:
- 각 클라이언트가 독립 배포 사이클
- Admin 코드를 User 사용자 노출 없이 격리
- AI 컨텍스트 분리로 토큰/품질 유리
- 2인이 각자 도메인 담당 가능
- 백엔드는 단일 (도메인 로직 재사용)

**왜 백엔드는 단일인가**:
- 빠른 개발 (4주 일정)
- 도메인 로직 재사용 (Customer/Deal 같은 도메인은 User/Admin 모두 사용)
- DB 단일 → 데이터 조회 일관성
- 분리 필요 시 `modules/admin/` 폴더만 떼면 됨 (분리 가능성 유지)

**공통 타입 공유 방법**:
1. 백엔드에서 OpenAPI 스펙 자동 생성 (`@nestjs/swagger`)
2. `sales-shared-types` 레포에서 `openapi-typescript`로 타입 추출
3. npm에 publish (private package) 또는 GitHub Packages
4. 웹/모바일이 `npm install @yourorg/sales-types` 로 사용

**대안 (간단)**: 초기엔 `AGENT/api-contract.md`에 수동 정의 → Phase 2에 자동화

### 3.5 인증 흐름 (4 클라이언트 공통)

```
[User Web/iOS/Android/Admin Web] ── 회원가입/로그인 ─► [Supabase Auth]
                                                            │
                                                            │ JWT 발급
                                                            │ (role 포함)
                                                            ▼
[Client] ◄────── access + refresh 토큰 ──────────────────────
    │
    │ 보관:
    │  - User Web / Admin Web: httpOnly cookie 또는 secure storage
    │  - Mobile: expo-secure-store (Keychain/Keystore)
    │
    │ Authorization: Bearer <access>
    ▼
[Nest API]
    │
    │ /api/*           → JwtAuthGuard (role 무관)
    │ /admin/api/*     → JwtAuthGuard + AdminGuard (role === 'ADMIN')
    │
    │ Supabase JWT secret으로 verify
    │ 통과 시 req.user = { id, email, role, ... }
    ▼
[Application Layer] ── 권한 검증, 비즈니스 로직
    │
    ▼
[Domain Layer] ── 비즈니스 로직 실행
```

**Admin 로그인 추가 검증**:
- Admin 웹은 로그인 응답의 `user.role !== 'ADMIN'`이면 즉시 로그아웃
- 라우트 가드(`AdminProtectedRoute`)에서 한 번 더 검증
- 백엔드 AdminGuard가 마지막 방어선

## 7. 모바일 앱 핵심 의사결정

### 7.1 EAS vs 자체 빌드
**결정**: EAS Build 사용

**근거**:
- 로컬에 Xcode/Android Studio 셋업 부담 없음
- iOS는 macOS 필수인데 EAS가 클라우드에서 처리
- 자동 코드사이닝 (인증서 관리 자동)
- 무료 티어 월 30회 충분

### 7.2 Expo Managed vs Bare Workflow
**결정**: Managed Workflow + 필요 시 Config Plugin

**근거**:
- Managed가 훨씬 간단
- 거의 모든 네이티브 기능을 Expo SDK로 커버 가능
- 정 필요하면 Config Plugin으로 네이티브 코드 주입

### 7.3 네비게이션: Expo Router vs React Navigation
**결정**: **Expo Router** (파일 기반)

**근거**:
- Next.js 같은 파일 기반 라우팅 → 웹 팀에 친숙
- Expo 공식 권장
- 딥링크 자동 처리

### 7.4 오프라인 저장소
**결정**: **expo-sqlite** (조회 캐시 + 변경 큐)

**근거**:
- AsyncStorage는 KV만 → 영업 데이터 쿼리 부적합
- expo-sqlite는 SQL 쿼리 → 필터링 그대로 적용 가능
- TanStack Query persistor와 결합 가능

### 7.5 상태 관리
**결정**: TanStack Query + Zustand (필요 시)

**근거**:
- 서버 상태는 TanStack Query (웹과 동일)
- 클라이언트 상태가 복잡해지면 Zustand
- Redux는 오버킬

### 7.6 UI 라이브러리
**결정**: NativeWind (Tailwind RN 포팅) + 자체 컴포넌트

**근거**:
- 웹 Tailwind와 같은 스타일링 → 인지 비용 절감
- shadcn/ui 모바일 버전은 아직 미성숙 → 자체 빌드

## 8. 백엔드 아키텍처 핵심 (자세한 건 AGENT/Backend/architecture.md)

### 8.1 Modular Monolith + DDD + Clean Architecture

**핵심**: 각 도메인 모듈이 자체 4계층을 가짐.

```
src/
├── modules/                    # 도메인별 모듈 (각자 4계층)
│   ├── user/                   # User 도메인
│   │   ├── domain/
│   │   ├── application/
│   │   ├── infrastructure/
│   │   └── presentation/
│   │
│   ├── auth/                   # 인증
│   ├── customer/               # 거래처
│   ├── deal/                   # 영업건
│   ├── company/                # 회사
│   ├── note/                   # off-the-record 메모
│   ├── business-card/          # 명함 스캔
│   │
│   └── admin/                  # ★ Admin 전용 도메인 (분리 대비)
│       ├── domain/
│       ├── application/        # 사용자 관리, 통계 등
│       ├── infrastructure/
│       └── presentation/       # /admin/api/* 라우트
│
├── shared/                     # 공통 (kernel)
│   ├── domain/
│   ├── application/
│   ├── infrastructure/
│   └── presentation/
│       ├── guards/
│       │   ├── jwt-auth.guard.ts
│       │   └── admin.guard.ts  # ★ Role === 'ADMIN' 체크
│       └── decorators/
│
└── main.ts
```

각 도메인 모듈 내부 (Clean Architecture 4계층):
```
modules/customer/
├── domain/                     # 순수 비즈니스 (외부 의존 X)
│   ├── entities/
│   ├── value-objects/
│   ├── events/
│   └── repositories/           # 인터페이스만
│
├── application/                # 유스케이스
│   ├── commands/
│   ├── queries/
│   ├── handlers/
│   └── services/               # findByUser(), findAllForAdmin() 둘 다 가능
│
├── infrastructure/             # 외부 시스템 어댑터
│   ├── persistence/            # Prisma 구현
│   ├── http/
│   └── messaging/
│
└── presentation/               # NestJS Controller
    ├── controllers/
    │   ├── customer.controller.ts        # User용 /api/customers
    │   └── admin-customer.controller.ts  # Admin용 /admin/api/customers
    └── dto/
```

**원칙**:
- 각 도메인 모듈은 **자체 4계층**(domain/application/infrastructure/presentation)
- Service Layer(Application)는 반드시 Domain Entity/VO를 통해 작업
- Repository는 인터페이스 (Domain) + 구현 (Infrastructure) 분리
- Domain은 Prisma 같은 외부 라이브러리 import 금지
- Controller는 thin (DTO 변환만)
- User Controller는 `/api/*`, Admin Controller는 `/admin/api/*` + AdminGuard
- Admin 전용 메서드는 이름에 명시 (`findAllForAdmin`)
