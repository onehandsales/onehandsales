# MVP 안정화를 먼저 해야 하는 이유

작성일: 2026-07-06

이 문서는 `onehand.sales`에서 새 기능을 추가하기 전에 MVP 안정화를 먼저 해야 하는 이유와 우선순위 판단 기준을 정리한다.

## 1. 결론

현재 단계에서는 새 기능 추가보다 MVP 안정화가 먼저다.

아직 사용자에게 공개하지 않았기 때문에 더더욱 안정화가 중요하다. 공개 전은 핵심 흐름, 데이터 안정성, UX/UI 혼란을 가장 싸게 고칠 수 있는 시점이다.

지금 목표는 기능이 많은 제품이 아니라, 처음 써본 사용자가 핵심 가치를 이해하고 한 번의 영업 업무를 끝까지 성공할 수 있는 제품을 만드는 것이다.

## 2. 현재 제품의 핵심 가치

`onehand.sales`의 현재 핵심 가치는 다음 흐름이다.

```text
영업 담당자가 회사, 담당자, 상품, 딜, 일정, 회의록을 한 곳에서 관리한다.
```

따라서 1차 MVP에서 반드시 안정적이어야 하는 흐름은 다음이다.

```text
로그인
→ 회사 등록
→ 담당자 등록
→ 상품 등록
→ 딜 생성
→ 일정 등록
→ 회의록 기록
→ 검색
→ 삭제
→ 휴지통 복구
→ XLSX 다운로드
```

이 흐름이 불안정하면, 태그, 리포트, 고급 필터, 자동화 같은 새 기능을 추가해도 사용자는 제품을 신뢰하기 어렵다.

## 3. 왜 공개 전 안정화가 중요한가

### 3.1 첫 사용자는 기능 수보다 신뢰감을 본다

CRM/영업 관리 도구는 사용자가 자신의 고객 정보, 연락처, 딜, 일정, 회의 기록을 입력해야 하는 제품이다.

사용자가 아래와 같이 느끼면 데이터를 넣지 않는다.

- 저장된 게 맞는지 모르겠다.
- 수정한 값이 제대로 반영됐는지 모르겠다.
- 삭제하면 복구되는지 불안하다.
- 딜과 담당자 연결이 맞는지 헷갈린다.
- 화면에서 다음에 뭘 해야 할지 모르겠다.
- 모바일에서 쓰기 불편하다.

초기 사용자는 기능이 적어도 핵심 흐름이 명확하고 안정적이면 계속 써볼 수 있다. 반대로 기능이 많아도 핵심 흐름이 불안하면 이탈한다.

### 3.2 공개 후에는 고치는 비용이 커진다

공개 후에는 실제 사용자 데이터가 쌓인다. 그 뒤에 데이터 구조, 삭제/복구 정책, import/export 정책, 인증 흐름을 고치면 비용이 커진다.

공개 전에는 다음 문제를 비교적 쉽게 고칠 수 있다.

- 데이터가 잘못 저장되는 문제
- 관계 데이터가 꼬이는 문제
- 삭제/복구 정책 문제
- 화면 구조 문제
- 문구와 UX 흐름 문제
- API 계약과 FE 사용 방식 불일치

공개 후에는 같은 수정도 마이그레이션, 데이터 보정, 사용자 안내, 운영 대응까지 필요해질 수 있다.

### 3.3 안정화 없이 새 기능을 붙이면 복잡도가 빠르게 커진다

태그, 리포트, 고급 검색, 자동화 같은 기능은 기존 핵심 데이터 위에 쌓인다.

기존의 회사, 담당자, 상품, 딜, 일정, 회의록 흐름이 불안정한 상태에서 새 기능을 추가하면 다음 문제가 생긴다.

- 버그 원인을 찾기 어려워진다.
- 기존 UX 문제가 새 기능에도 복제된다.
- 데이터 관계가 더 복잡해진다.
- 테스트 범위가 커진다.
- 나중에 구조를 바꾸기 어려워진다.

새 기능은 핵심 흐름이 안정된 뒤에 붙여야 가치가 살아난다.

## 4. 지금 하지 않아야 하는 일

아래 작업은 지금 당장 우선순위가 아니다.

- 새 대형 기능 추가
- 관리자 운영 페이지 확장
- 알림 기능 구현
- 과금 및 구독 결제 구현
- 태그, 리포트, 자동화 같은 확장 기능 선구현
- 디자인 전체 리뉴얼부터 시작하기
- 기능 QA 없이 화면만 예쁘게 다듬기

디자인이 중요하지 않다는 뜻이 아니다. 다만 지금 필요한 디자인 작업은 전체 리브랜딩이 아니라, 핵심 업무 흐름을 더 쉽게 이해하고 완료하게 만드는 UX/UI 안정화다.

## 5. 지금 먼저 해야 하는 일

### 1순위. MVP 핵심 흐름 QA

`QA_CHECKLIST.md`를 기준으로 실제 사용 시나리오를 수행한다.

최우선 시나리오:

```text
로그인
→ 회사 생성
→ 담당자 생성
→ 상품 생성
→ 딜 생성
→ 일정 생성
→ 회의록 생성
→ 검색
→ XLSX 다운로드
→ 삭제
→ 휴지통 복구
```

이 시나리오를 데스크톱과 모바일에서 각각 확인한다.

### 2순위. S0~S2 버그 수정

QA 결과를 심각도로 나눈다.

- `S0 Blocker`: 앱 실행 불가, 로그인 불가, 핵심 화면 진입 불가
- `S1 Critical`: 데이터 손실, 권한 문제, 다른 사용자 데이터 노출
- `S2 Major`: 핵심 업무 기능 실패
- `S3 Minor`: 필터, 정렬, 일부 UX, 문구 문제
- `S4 Polish`: 시각적 완성도 개선

먼저 `S0~S2`만 고친다. 이 단계의 목표는 예쁜 제품이 아니라 업무가 끝까지 되는 제품이다.

### 3순위. 핵심 화면 UX/UI 안정화

기능이 통과한 뒤 사용 빈도가 높은 화면부터 UX/UI를 정리한다.

추천 순서:

1. 홈
2. 딜 목록/상세
3. 회사 목록/상세
4. 담당자 목록/상세
5. 일정
6. 회의록
7. Import
8. Trash

UX/UI QA의 기준:

- 사용자가 다음 행동을 바로 알 수 있는가?
- 목록에서 중요한 정보가 바로 보이는가?
- 저장, 삭제, 복구 후 피드백이 명확한가?
- 모바일에서 글자, 버튼, 모달이 깨지지 않는가?
- loading, empty, error 상태가 자연스러운가?
- 문구가 딱딱하거나 불안하게 느껴지지 않는가?

### 4순위. 운영 안정성 보완

핵심 흐름과 UX/UI가 어느 정도 안정되면 운영 안정성을 보완한다.

우선순위가 높은 항목:

- Import 확정 전 작업 DB 영속화
- `BE/.env.example` 추가
- `FE/user-web/.env.example` 추가
- 배포/로컬 실행 문서 정리
- 현재 구현 범위가 아닌 FE 코드 노출 여부 정리

### 5순위. 새 기능 고도화

그 다음에 새 기능을 검토한다.

추천 순서:

1. 검색/필터 고도화
2. 주간 일정/리포트
3. 회의록 AI/STT 이력 정책 정리
4. 휴지통 정책 고도화
5. 태그 기능
6. Export 경험 정리

## 6. MVP 안정화 완료 기준

아래 조건을 만족하면 MVP 안정화 1차 완료로 본다.

- BE `typecheck`, `lint`, `test`, `build` 통과
- FE/user-web `typecheck`, `lint`, `build`, `test:e2e` 통과
- 핵심 수동 시나리오 1회 이상 통과
- 회사/담당자/상품/딜/일정/회의록 CRUD 통과
- Search, Trash, Import, Export 기본 흐름 통과
- 모바일 390px 기준 핵심 시나리오 사용 가능
- S0/S1 버그 없음
- S2 버그가 출시 판단 가능한 수준으로 정리됨
- 알려진 한계와 제외 범위가 문서화됨

## 7. 의사결정 원칙

새 작업을 시작하기 전에 아래 질문에 답한다.

1. 이 작업이 핵심 업무 흐름을 안정화하는가?
2. 이 작업이 데이터 손실이나 권한 문제를 줄이는가?
3. 이 작업이 사용자가 다음 행동을 이해하는 데 직접 도움 되는가?
4. 이 작업이 공개 후 고치기 어려운 문제를 지금 줄이는가?
5. 이 작업이 단순히 기능 수를 늘리는 작업은 아닌가?

1~4 중 하나라도 강하게 해당하면 우선순위를 높인다. 5에만 해당하면 MVP 안정화 이후로 미룬다.

## 8. 이번 단계의 한 문장 목표

```text
첫 사용자가 로그인부터 딜 관리, 일정/회의록 기록, 검색, 삭제/복구까지 막힘 없이 끝낼 수 있게 만든다.
```
